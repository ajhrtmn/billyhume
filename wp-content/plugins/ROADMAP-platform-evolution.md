# ROADMAP: platform evolution — storefront, monetization depth, LMS builder, user portal, social

Companion to `VISION.md` (read that first for the whole ecosystem's architecture and shared services) and the other two roadmap docs. This one covers a distinct, later-stage ambition: growing this ecosystem from "a business-in-a-box built ON WordPress/WooCommerce" toward "a business-in-a-box that OWNS its own foundations, with WordPress/WooCommerce as swappable implementation details rather than load-bearing assumptions." It also covers the four concrete feature areas that prompted this doc: a real storefront/merchandising layer, deeper monetization tiers, a proper LMS course/lesson builder, and a fully custom user-facing account portal — plus a roadmap-only look at social/marketing.

This doc assumes the reader has also read `QA-REPORT.md` from the most recent QA/refactor pass — several of "what's next" items below build directly on fixes and gaps that pass surfaced.

## 0.-a. Standing priority override: harden before extending

Per the project owner's explicit current direction (mirrored in full in `VISION.md`'s "Current standing priority" section — read that version for the complete rationale): everything in this doc is paused, conceptually, behind a core-hardening pass covering `own-ur-shit`, `bh-contest`, the `BHY_*` style system, `bh-crm`, and Debug Tools. Concretely this means jobs/errors/logs/testing/queue infrastructure, the abstraction boundaries this doc itself proposes (`BH_Content`, `BH_Commerce`, etc.), and cross-plugin load-order/third-party-conflict issues get fixed before any new storefront/LMS-builder/portal/social work below gets picked up. None of the design thinking in this doc is invalidated by that — it's sequencing, not scope-cutting.

## 0. The honest framing, stated up front

"Eventually outgrow the need for WordPress, WooCommerce, or any 3rd-party solution" is a genuinely large claim, and it deserves a genuinely honest answer rather than either dismissing it or overpromising it. WordPress itself is not one dependency — it's dozens: the user table and session/auth model, the post/postmeta storage layer, the media library, `$wpdb`, WP-Cron, the whole plugin-loading and hook system, REST API routing, and (the newest addition to this list) Gutenberg's block model. Replacing all of that is closer in scope to building a competing application framework than to a feature project — realistically a multi-year undertaking if it were ever fully pursued, not something this roadmap can respect and also pretend fits in a quarter.

What IS realistic, and what this whole doc is actually organized around: **build our own internal interfaces now, so every 3rd-party system is a plug-in behind that interface rather than something scattered directly through the codebase.** That doesn't mean writing our own commerce engine tomorrow — it means that `BHM_Products` (say) stops calling `wc_get_product()` directly and instead calls `BH_Commerce::get_product()`, where `BH_Commerce` is our own interface that WooCommerce happens to implement today. The switching cost of "replace WooCommerce" drops from "rewrite every call site across six plugins" to "write one new implementation of one interface" — which is the actual, achievable version of the ambition, and the reason the interface-first posture was chosen for this phase (see Section 2).

**The same "remove third parties wherever realistically possible" instinct now extends past this ecosystem's own dependencies to a second, wider set of 3rd-party tools entirely** — BI/reporting (SQL Server Reporting Services/Report Builder, actively being sunset by Microsoft in favor of a paid Power BI tier), ERP/accounting (NetSuite, and the smaller QuickBooks/Xero tier below it), individual + small-business taxes, and email/calendar (Gmail/Google Workspace, Outlook/Microsoft 365) — added as a second round of "big-vision pillars" in `VISION.md` (see that doc's own "round two" section for the full, honestly-scoped accounting of each, including which ones are realistic near-term builds vs. multi-year regulatory undertakings). Not detailed further in this doc, which stays scoped to the music-business vertical's own platform evolution — flagged here only so the two docs don't drift out of sync about how big the "own the interfaces" ambition has actually gotten.

## 1. What already exists, mapped against the new asks

Reading this ecosystem's current code against the five feature areas below, to avoid re-building what's already there:

- **Identity/auth** (`BHI_Auth`, `BHI_Profiles`) is real and reusable — registration, login, session, email verification, a public profile page. What it does NOT have yet: an authenticated "my account" dashboard beyond the single public profile page. This is the foundation the new user portal builds on, not a rewrite.
- **Design tokens** (`BHY_Style` for front-end, `BHY_UI` for wp-admin, a live Style Gallery) exist and are already consumed by every peer plugin's front-end rendering. This is real design-system infrastructure, but it's a shared stylesheet + component-class library, not a content/layout BUILDER — there's no drag-drop, no nested-block authoring, no visual composition tool anywhere in the codebase today. That gap is real and is Section 3's subject.
- **Monetization** (`BHM_Tiers`, `BHM_Gate`, `BHM_Wallet`, `BHM_Products`) is a working Patreon-lite: supporter tiers backed by real WooCommerce products (subscription or one-time depending on whether WC Subscriptions is active), a generic paywall gate any content type can call, a pay-per-play wallet with an authoritative ledger, and (as of the QA pass) refund/fraud-pattern detection split into its own `BHM_Fraud` class. What it's missing against Patreon's actual current feature set (see Section 4): per-tier structured benefit lists (today a tier is just a name + price), tier cover images, annual pricing with a discount incentive, gifting/free trials/promo codes, and any tier-level integration hooks (Patreon's Discord-role-per-tier being the concrete example worth having an equivalent extension point for, even if Discord itself isn't built yet).
- **LMS** (`BHC_PostTypes`, `BHC_Steps`, `BHC_Progress`, `BHC_Gate`) has real, working course → lesson → step structure, quiz scoring, progress tracking, and drip scheduling, already wired to `BHM_Gate` for tier-gating. What it's missing: the actual AUTHORING experience is a fixed-shape metabox form (a JSON array of steps, each one of four hardcoded types: text/image/video/quiz) — not a flexible, nested, visually-composed lesson. This is where the content-block builder from Section 3 plugs in most directly.
- **Commerce** is 100% WooCommerce today, called directly from `bh-monetization-woo` with no abstraction in between. There is no storefront/browse/collections/merchandising layer at all — an artist selling tracks/releases/tiers today gets whatever WooCommerce's stock shop/archive templates render, which is generic and not tailored to "browsing an artist's catalog." This is Section 5's subject, and also the area most worth routing through the new Commerce interface first, since it's simultaneously the most WooCommerce-coupled area AND the best real test of whether that interface is actually well-designed.
- **User-facing portal**: doesn't exist as a concept yet. Contest submission management lives inside `bh-contest`'s own front-end, subscription/wallet management lives inside `bh-monetization-woo`'s shortcodes, LMS progress lives inside `bh-courses` — each plugin renders its own front-end pieces independently, with no single "this is your account" surface tying them together. Section 6 covers building that shell.
- **wp-admin exposure**: real users (subscriber-role accounts created via `BHI_Auth::register()`) are never given admin capabilities today, and the whole ecosystem's own convention is that end-user actions go through `admin-post.php` handlers or REST routes, never through an admin-side UI — so there's no NEW security hole to close here, just a UI-shell gap (nothing currently stops a curious subscriber from typing `/wp-admin` into their address bar and seeing WordPress's bare default dashboard, which is confusing and off-brand even though it grants them nothing).

## 2. The abstraction layer — build these interfaces now (interface-first, per direction given)

Four internal interfaces, each with a WordPress/WooCommerce-backed implementation shipping now and a documented contract that a future non-WordPress implementation could satisfy later. All four live in the core (`own-ur-shit`) next to the existing shared services (`OUS_Notifications`, `OUS_Jobs`, etc.) — same "one shared service, zero central registration needed by consumers" shape this ecosystem already uses successfully.

1. **`BH_Content` (content-block interface)** — the highest-priority one, since the LMS lesson builder, the storefront's merchandised landing pages, and (eventually) the portal's own customizable panels all need to render and author nested, structured content. Contract: a content document is a tree of typed blocks (`{type, attrs, children}`), each type registered once (`BH_Content::register_block_type($type, $schema, $renderer)`) by whichever plugin owns it — a quiz-question block from bh-courses, a tier-comparison block from bh-monetization-woo, a track-embed block from bh-streaming — and the interface handles serialization, validation against the type's schema, and rendering. The WordPress-backed implementation underneath can literally BE Gutenberg's own block model for post-attached content (reusing `post_content`'s existing JSON-comment block format costs nothing and stays compatible with core WP tooling), while non-post contexts (a lesson step, a tier's benefit list) use the same block-tree shape stored in a plain JSON column instead of `post_content`. This is the layer a future GrapesJS-style visual editor gets built on top of — the editor is a front-end concern; the interface underneath doesn't change whether the authoring UI is a form, a drag-drop canvas, or (later) something we own outright.
2. **`BH_Commerce` (commerce interface)** — wraps product CRUD, cart, checkout, and order/subscription status behind our own contract. `BHM_Tiers::save()`/`sync_tier_wc_product()` and `BHM_Products`' object-purchase-product sync are the concrete existing call sites that move behind this first — they're already isolated enough (per the QA pass's SRP work on `class-products.php`) that this is a mechanical, well-scoped migration, not a rewrite. Payment GATEWAYS (Stripe, PayPal, etc.) stay behind WooCommerce's own gateway API for now — this interface is about not being coupled to WooCommerce's cart/order/product model specifically, not about reimplementing payment processing, which is its own, much later, much larger question.
3. **`BH_Media`(storage/delivery interface)** — thinner than the other two since the ecosystem already goes through one API surface (`wp_get_attachment_url()` and friends) for all media, which is exactly why Advanced Media Offloader already works with zero code changes today. Worth a formal interface anyway so a future non-WP media store isn't a surprise migration — but genuinely low-priority relative to the other three given how little is actually WP-coupled here already.
4. **`BH_Mail` (transactional delivery interface)** — not asked for explicitly, but surfaced by this analysis: `OUS_Notifications`' email channel currently rides on `wp_mail()`, which is notoriously unreliable without a real ESP configured underneath (no delivery guarantees, easy to land in spam, no bounce/complaint handling) — this is a hidden 3rd-party-shaped dependency (on whatever mail transport the server happens to have) that's worth abstracting for the same reason as the other three, and worth prioritizing once real subscription/purchase-confirmation email volume matters.

**What does NOT get a new interface yet:** the user/session/auth table (`BHI_Auth` already sits on top of WP's `wp_users`/`wp_usermeta` and cookie-auth, and re-platforming identity is a much bigger, much riskier undertaking than the four above — correctly out of scope for this phase), and WP-Cron/the job queue (`OUS_Jobs` already treats WP-Cron as an implementation detail behind its own `enqueue()`/`register()` contract, so this one's arguably already done).

## 3. LMS course/lesson builder & the general block-composition tool

This is the single highest-leverage piece, exactly as flagged going in — it's the foundation both the LMS authoring experience and the storefront's landing/collection pages actually need, and probably the closest thing in this whole roadmap to "infrastructure other future things get built on."

**Shape, informed by the reference points named (GrapesJS, a HubSpot-CMS-inspector-style structured-data view, phpMyAdmin/cPanel-style raw data access, Storybook's presentation of a component library):**
- A visual, nested-block canvas (GrapesJS is a legitimate, mature, MIT-licensed open-source library for exactly this — drag/drop composition, a style manager, a layer/outline panel, code export — and is a reasonable base to build the authoring UI on top of `BH_Content` rather than inventing a canvas engine from scratch).
- A component/block "catalog" view in the Storybook spirit — this ecosystem's own Style Gallery (`bhy_style_surfaces`) already does a lighter version of exactly this for whole rendered surfaces; the block-builder's own catalog is the natural next evolution of that same idea at the individual-block level, and should probably absorb/supersede the Style Gallery over time rather than becoming a second, competing gallery concept.
- A structured-data table view (the "HubDB/phpMyAdmin" reference) for the block TYPES that are genuinely tabular rather than free-form layout — a course's list of lessons, a tier's list of benefits, a product's list of variants. This is a real, distinct UI mode from the visual canvas (spreadsheet-like row/column editing beats drag-drop for "reorder 40 quiz questions"), and both modes should edit the exact same underlying `BH_Content` block tree rather than being two disconnected editors.
- The "layer over standard Gutenberg, customized to work how we want" option is worth taking seriously as the FIRST implementation rather than a fallback: Gutenberg already solved block registration, nested block trees, undo/history, and collaborative-editing-adjacent infrastructure; building `BH_Content`'s WordPress-backed implementation as a thin, opinionated layer on top of the existing block editor (custom block types, a curated/simplified inserter, our own design-token-driven style controls instead of Gutenberg's generic ones) is very plausibly less total work than a from-scratch GrapesJS integration, while still leaving the door open to swap the AUTHORING UI later (GrapesJS, or something fully custom) without touching `BH_Content`'s contract or every block type's renderer.

**Concrete LMS payoff once this exists:** lesson authoring stops being a fixed four-step-type JSON form and becomes real nested composition — a lesson can mix any registered block types in any order/nesting (text explaining a concept, an embedded audio example via a bh-streaming track-embed block, an image with caption, a quiz block, a "compare to a past submission" block for BH Feedback later), authored visually instead of through a metabox form. Progress tracking (`BHC_Progress`) doesn't need to change shape — it already tracks per-step completion by index; a "step" just becomes "one top-level block in the lesson's content tree" instead of one entry in a fixed-type array.

## 4. Monetization tier depth (Patreon-equivalent feature parity)

Current-state Patreon feature research (July 2026) to target against, alongside what this ecosystem already has:

- **Structured per-tier benefits list + tier cover image** — Patreon tiers show a name, cover image, and an explicit bulleted benefits list. `BHM_Tiers` today stores a name and price only; add a benefits list (a natural `BH_Content` block-list once Section 3 exists — reuse rather than inventing a second rich-text field type) and a cover image (already trivial, same attachment pattern used everywhere else in this ecosystem).
- **Annual pricing with a discount incentive** — Patreon creators offering annual plans earn meaningfully more on average and see roughly double the retention of monthly members; `BHM_Tiers`/`sync_tier_wc_product()` currently hardcodes `subscription_period => 'month'`. Adding an annual option (a second WC Subscription product per tier, or WC Subscriptions' own variable-subscription support if that fits better) is a concrete, well-scoped extension of existing code, not new architecture.
- **Gifting, free trials, discount promo codes** — none of these exist today. WooCommerce Subscriptions has native support for trials and (via extensions or WooCommerce's own coupon system) discounts; gifting a membership is closer to "buy a tier on someone else's behalf," which is a real product-flow addition (recipient email capture at checkout, a claim/redemption flow) rather than a WooCommerce-native feature — worth scoping as its own small project once the tier-benefits/annual work above lands.
- **Tier-level third-party integration hook** — Patreon's "connect Discord and grant a role per tier" is the concrete example. Not proposing Discord integration be built now (that's squarely Section 7/roadmap-only territory), but `BHM_Tiers`/`grant_entitlement()` should expose a `bhm_entitlement_granted`/`bhm_entitlement_revoked` action pair (a small, cheap addition, and arguably already 90% implied by the existing `OUS_Notifications`-integrated `grant_entitlement()` choke point from the QA pass) so a future Discord-role-sync feature — or any other tier-gated external system — has one clean place to hook in rather than needing to modify `BHM_Products` directly.
- **The LMS↔monetization tie-in already asked for** ("let the LMS tap into that") is architecturally already true today — `BHC_Gate::user_can_access_course()` already calls `BHM_Gate::user_has_tier_access()` when bh-monetization-woo is active, exactly the generic-paywall pattern `BHM_Gate`'s own docblock anticipated. Nothing new needed here beyond the general tier-depth work above flowing through automatically (a richer tier with benefits/annual pricing gates a course exactly the same way a plain tier does today).

## 5. Storefront & merchandising

The most WooCommerce-coupled area today (bare shortcode-rendered tier/tip/wallet blocks, otherwise whatever WooCommerce's stock templates produce) and the best real proving ground for the `BH_Commerce` interface from Section 2, since building this ON that interface from day one (rather than directly on `wc_get_products()`/WooCommerce query classes) is both the actual point of building the interface and a concrete way to validate it's well-shaped before more code depends on it.

Feature targets, drawing on Amazon (deep faceted filtering, merchandising rules, related/"customers also bought"), Apple (restraint — a small number of genuinely well-designed collection/product templates rather than infinite configurability), and Shopify (the actual current bar for "an independent seller's storefront," particularly collection pages, product page layout conventions, and its own theming/section model worth studying as prior art for how a merchant-facing customization layer should feel):

- **Collections/categories as first-class, richly composed pages** — not a generic WooCommerce category archive, but a `BH_Content`-composed landing page per collection (an artist's whole catalog, a specific release, a "supporter exclusives" collection) mixing merchandising blocks (featured item, editorial copy, a filtered grid) the same way any other `BH_Content` page does — this is where Section 3's builder pays for itself a second time.
- **Individual listing/product pages** with the same treatment — track/release/tier/course pages composed from blocks rather than a fixed WooCommerce single-product template, while still checking out through the real `BH_Commerce`-wrapped WooCommerce cart underneath.
- **Browse/search/filter controls** — faceted filtering (genre, price, tier-gated vs. open, release date) is a real, somewhat involved feature (needs either WooCommerce's product-attribute/taxonomy system leaned on harder than it is today, or a dedicated search index if the catalog ever gets large) — scope this as its own project once collections/listing pages exist, not bundled into the first pass.
- **Merchandising/relations** ("customers also bought," "more from this artist") — bh-streaming's own `BHS_Recommendations` (content-based, same-artist/same-release/shared-genre scoring, just fixed in the QA pass) is directly reusable prior art for the SAME kind of relation on the commerce side; worth sharing the scoring approach rather than inventing a second one.

### 5a. Auction listings (new addition — roadmap only, not yet scoped for a build pass)

A time-boxed bidding mechanism, layered onto this section's own `BH_Commerce`-wrapped WooCommerce plumbing rather than a parallel payments/checkout stack — the same "own the interface, lean on WooCommerce underneath" posture Section 5 already establishes for the rest of the storefront. Natural fit for one-of-a-kind merch, signed items, limited-run physical releases, or a fan-funded "name your price with a floor" mechanic genuinely distinct from the flat-price tier/purchase model `bh-monetization-woo` already has.

Real open questions to resolve before this gets a real design pass, named here so they're not lost:
- **Payment authorization timing.** A bid likely needs a hold/authorize-not-capture flow (don't charge until the auction closes and this bid wins) — does this want a `BHM_Wallet`-style credit hold, or a real payment-gateway-level authorization hold via WooCommerce's own payment-gateway API? The former keeps everything self-hosted/in this ecosystem's own ledger; the latter is more "how eBay/real auction sites actually work" but reintroduces a closer coupling to whatever payment gateway is active — worth weighing against this doc's own Section 0 "own the interfaces" posture explicitly when this gets designed for real.
- **Post type / data model.** Does an auction listing want its own CPT, or reuse the existing product/tier machinery with a new "auction" pricing mode (a start price, minimum increment, close datetime, optional reserve, optional buy-it-now)? The latter is more consistent with how tiers/purchases already share machinery in `class-tiers.php`; the former may be cleaner given bidding has its own lifecycle (open → closing → closed → fulfilled) that doesn't map cleanly onto a simple product purchase.
- **Fraud/abuse surface.** Bidding introduces a new abuse pattern (shill bidding, bid retraction abuse, last-second sniping) that `BHM_Fraud`'s existing refund/chargeback-pattern detection doesn't cover today — see `ROADMAP-safety-and-metrics.md` for where this kind of thing should eventually get designed alongside the existing fraud work, not as a separate bolt-on system.
- **Closing mechanism on WP-Cron.** An auction closing at an exact time is a real-time-sensitive operation running on a traffic-dependent cron (same honest caveat this ecosystem's `OUS_Jobs` docblock already gives every time-sensitive job) — needs a close-and-settle job that tolerates running a few minutes late gracefully (e.g., accept any bid placed before the nominal close time even if the settlement job itself runs slightly after) rather than assuming cron fires exactly on time.

Best sequenced after the rest of Section 5 (collections/listings/browse) exists, since an auction listing is a specialized listing TYPE and benefits from the same `BH_Content`-composed page treatment the rest of the storefront gets.

## 6. Custom user-facing portal (full build, per direction given)

A genuinely separate, branded front-end account area — not a reskinned wp-admin, not a collection of independent per-plugin shortcodes on separate pages. Concretely:

- **`BHI_Portal`** (new core class) renders one authenticated account shell (a page template or a top-level rewrite-rule-owned URL, e.g. `/account/`) with its own navigation, built from panels contributed by any plugin via a new `bhi_portal_panels` filter — the same zero-central-registration shape `ous_registered_plugins`/`ous_debug_tools`/`bhy_style_surfaces` already use successfully. Each panel entry: a label, an icon, a render callback, and (reusing the existing convention) an optional `class_exists()`-guarded contribution so panels the visitor's active plugins don't cover simply don't appear.
- **Concrete panels for what already exists:** profile/identity (already built, just needs to move from a standalone public-profile page into a portal panel — the PUBLIC profile page stays, since that's a different, intentionally-shareable surface), subscription/tier + wallet management (bh-monetization-woo), course progress/"continue learning" (bh-courses — directly reuses `BHC_Progress`), contest submission management (bh-contest — a new capability, since submissions today are presumably create-once via the contest's own front end; "edit your submission" is a real feature gap this portal build should close, not just a UI move), and notifications inbox (already exists as `[bh_notifications]`, becomes a portal panel instead of/alongside a standalone shortcode).
- **wp-admin exclusion, confirmed as a solved problem (current WordPress practice, verified July 2026):** redirect any non-elevated role away from `/wp-admin` entirely on `admin_init` (not just hiding the admin bar, which is cosmetic and leaves the dashboard reachable by direct URL), and disable `show_admin_bar` for the same roles — both are well-established, low-risk patterns, not a security research problem. This ecosystem's own existing convention already supports this cleanly: every end-user-facing write already goes through `admin-post.php` handlers or REST routes rather than an admin-side screen (see `BHI_PublicProfile::handle_save()` as the existing pattern), so blocking `/wp-admin` entirely for subscriber-role accounts costs nothing functionally — no existing feature routes a real user through wp-admin today. Roll this out ecosystem-wide as part of the portal build, in the core (one place, one filter for role exemptions), not per-plugin.

## 7. Social & marketing — roadmap only this phase (no build tasks yet, per direction given)

Current-state platform research (July 2026), to ground this section in what's actually available rather than what was true as of the ecosystem's own knowledge cutoff:

- **YouTube Data API v3** — genuinely accessible: no monetary cost, a default 10,000-unit daily quota, and a recent (~Dec 2025) quota-cost reduction plus a further change moving `videos.insert` to its own dedicated ~100-call daily bucket separate from the shared quota — meaning automated video upload/cross-post is now realistically cheap to build against, a meaningfully better starting position than it was a year ago. **Read for accuracy, not assumed** — verify current quota mechanics against Google's live quota calculator before scoping real work here, since this specific area changed materially and recently.
- **Twitch API (Helix + EventSub)** — mature, well-documented, actively maintained; EventSub (webhook or WebSocket delivery of stream-live/follow/etc. events) is the natural integration point for "announce when this artist goes live" cross-promotion, and Conduits exist for higher-scale event delivery if that's ever needed.
- **Meta Graph API (Instagram/Facebook)** — the most gated of the four: requires a Business/Creator Instagram account linked to a Facebook Page, a Meta developer app, and separate app-review submissions (each with a screencast) per permission scope (`instagram_business_basic`, `instagram_business_content_publish`), with 2-4 week review turnaround per submission, plus a 100-published-posts-per-24-hours cap. Budget real calendar time for the review process if this is ever built, not just engineering time.
- **TikTok Content Posting API** — similarly gated: developer registration, app review (a demo video/screenshots plus a working "Post to TikTok" consent screen showing privacy/Duet/Stitch settings is a REQUIREMENT of the review, not just a nice-to-have), roughly 5-10 business day review turnaround, and a development-mode sandbox limited to private-only posts and 5 authorized accounts per 24 hours until approved.
- **Roku Ads (advertising, distinct from the above four which are organic/cross-posting platforms)** — genuinely relevant and worth having researched given it was named specifically: Roku Ads Manager (self-serve, ~$500 minimum campaign spend, audience targeting/budget control/conversion tracking) is the accessible entry point for an independent artist; Roku OneView is the programmatic/agency DSP side with much higher minimums (~$25,000+) and isn't the right fit for this audience. If paid CTV advertising is ever built into this ecosystem, Ads Manager is the one to integrate against, not OneView.

**The stated emphasis — organic reach and word-of-mouth over paid acquisition, and support for physical/in-person experiences — deserves a concrete idea, not just a values statement, even while staying roadmap-only:** the portal (Section 6) and the block-builder (Section 3) are both prerequisites for the most honest version of "social integration" this ecosystem could build — not a firehose of auto-cross-posted content, but tools that make a fan's own organic sharing easy (a well-composed, block-built public profile/release page that's actually good to share, a "who's going" / RSVP block for a real show or listening party, referral-aware share links feeding the CRM's existing `bh_crm_activity_summary` pattern) — worth keeping as the guiding frame when this section eventually graduates from roadmap to build, rather than defaulting to "add auto-poster integrations for four platforms" as the first move.

## 7a. In-admin version history for user-built content (new addition — roadmap only, NOT YET SCOPED, not started)

Captures a feature idea from the project owner, verbatim intent preserved: "add in admin version control for pretty much everything eventually" — clarified as "By in-admin I mean in the WP dashboard, and by everything, I mean anything we can think of the users might build and want multiple versions of as they are building them." This is explicitly NOT git-based and NOT developer-facing — it's an in-wp-admin, WordPress-core-post-revisions-style history/restore UI, generalized to the admin-built objects across this ecosystem that don't use `wp_posts` (or use it inconsistently), so a user iterating on something can see, save, and restore prior versions the same way core already does for a post.

Candidate targets, named here so the scope is concrete rather than "everything": CRM records/notes (bh-crm), style/theme configs (`BHY_UI`), LMS courses/lessons/quizzes (bh-courses, and directly relevant once Section 3's `BH_Content` block-tree lesson authoring exists), contest configurations (bh-contest), portal layouts (`BHI_Portal`, Section 6), and the new visual "element builder" for styleable containers/widgets requested elsewhere this session.

Likely technical shape — an open question, not a decision: could ride the existing `bhcore_events`/job-queue infra for storage, or use a dedicated lightweight revisions table keyed by object-type + object-id + version. Either way it should probably be one shared service (same "one shared service, zero central registration needed by consumers" shape `OUS_Notifications`/`OUS_Jobs` already use) rather than each plugin inventing its own versioning.

**Status: NOT YET SCOPED, NOT STARTED.** No design pass, no data model decision, no code — this section exists only to capture the idea for a future session.
