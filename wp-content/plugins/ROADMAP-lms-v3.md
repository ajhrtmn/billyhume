# ROADMAP: LMS v3 — interactive video, richer authoring, branching paths, mind-map authoring

Companion to `LMS-AUTHORING-DESIGN-PLAN.md` (the prior design pass this one picks up from) and `ROADMAP-platform-evolution.md` Section 3 (the block-composition tool this doc's ideas all sit on top of). Scopes four bigger LMS ambitions raised with Duolingo, Coursera, Khan Academy, Udemy, and Unity's training content named as reference points, against the LMS as it actually exists today in `bh-courses` — not implemented yet, roadmap and scoping only.

## 0. The honest framing, stated up front

Three of these four ideas are real, buildable features that extend the existing architecture without breaking it. One of them — branching lesson paths — is not a feature extension, it's a change to the LMS's core data model, and it's worth being direct about that distinction now rather than discovering it mid-build. The mind-map tool is a different kind of large: not architecturally risky the way branching is, but closer to a new authoring surface than a feature, in the same sense Section 3 of `ROADMAP-platform-evolution.md` already treats the general block-composition canvas as its own project rather than a lesson-editor detail.

Grounding this in what's actually there today (not memory, not the design doc's stated intentions): `BHC_Progress::step_status()`/`mark_step_complete()` key everything on a plain `(user_id, lesson_id, step_index)` composite — `step_index` is an integer offset into an ordered array, full stop. `first_incomplete_lesson()` determines "what's next" by counting completed indices against `BHC_Steps::count($lesson_id)` — sequential completion counting, not graph traversal. Nothing in `class-progress.php` or `class-steps.php` has a concept of "next step depends on what you just did." `BH_Content`/`BH_Studio` (the block-authoring system bh-courses' lesson steps are being migrated onto per `LMS-AUTHORING-DESIGN-PLAN.md`) is a nested container/child tree — arbitrary nesting, but still strictly linear top-to-bottom in how it renders and how progress walks it. Neither system was designed with branching in mind, and neither has been extended toward it.

## 1. Interactive video overlays (Khan Academy / pop-up-question style)

**Realistic near-term build**, bounded by existing infrastructure rather than requiring new architecture.

What exists to build on: bh-courses already has a native `video` step type (`class-steps.php`, step type registry) supporting both `source: 'upload'` (attachment ID) and `source: 'url'` (external embed), sanitized in `save()`. bh-streaming's `BHS_Player` (`class-player.php`) is deliberately minimal — `init`/`maybe_enqueue`/`render`, no chapter markers, no overlay hooks, nothing to inherit from there. There is no annotation/overlay/interactive-video concept anywhere in either plugin today — this is genuinely greenfield on the UI/interaction side, but sits on a step type that already exists rather than needing a new one.

Shape worth targeting: a JSON list of timestamped annotations on the video step (`{ time, type: 'question' | 'hotspot' | 'note', payload }`), rendered by a small JS player wrapper (pause the native `<video>`/embed at the timestamp, show the overlay, resume on dismiss/answer). A `'question'` annotation type can reuse the quiz step's existing scoring/storage pattern (the same `answers` JSON snapshot shape just added to `bhc_progress` for regular quiz steps) rather than inventing a second scoring mechanism — genuinely additive to what Part 1 of the last pass already built, not a fork of it.

Real open question, not a blocker: does pausing mid-video for a question count as "one step" (the whole video, annotations included, is `step_index N`) or do individual annotations need their own completion tracking (a sub-index)? The first option fits the current linear model with zero schema change; the second edges toward the same non-linear-progress question Section 2 raises for branching, just at much smaller scale (an annotation redirect is bounded to "resume the same video," never "jump to a different lesson"). Worth deciding explicitly during implementation, but doesn't require a dedicated design pass first — the blast radius if the wrong call is made is one step type, not the whole progress model.

No Opus/Plan pass needed. This is routine extension of an existing step type once someone decides the annotation-vs-step granularity question above.

## 2. More robust course/lesson authoring tools generally

**Realistic near-term, ongoing** — this is largely `ROADMAP-platform-evolution.md` Section 3 (the general `BH_Content`/`BH_Studio` block-composition tool) continuing to mature, not a new initiative. The LMS-specific payoff was already named in that doc: "lesson authoring stops being a fixed four-step-type JSON form and becomes real nested composition... authored visually instead of through a metabox form" (Section 3), and `LMS-AUTHORING-DESIGN-PLAN.md` already wired the first real slice of that (`bhc/*` block types registered onto the Studio canvas, `bhc/quiz` promoted to a real container of `bhc/quiz-question` children).

Concretely still open, all incremental: more block types (the interactive-video block from Section 1 above becomes one once it exists), a better inserter/curated palette specific to LMS authoring rather than the generic Studio canvas, drag-reorder for lesson/step sequencing beyond what the metabox-replacement work already did, and the structured-data table view `ROADMAP-platform-evolution.md` Section 3 named ("a course's list of lessons... spreadsheet-like row/column editing beats drag-drop for reorder 40 quiz questions") — genuinely useful for a course with many quiz questions specifically, and not yet built anywhere.

No dedicated design pass needed beyond normal per-feature scoping as each piece gets picked up — this is the same "routine implementation once direction is set" category most of this session's work has been.

## 3. Branching lesson paths ("choose your own adventure")

**Needs a dedicated Opus/Plan-mode design pass before any code touches this.** Flagging explicitly, per standing instruction — this is exactly the kind of real multi-file architecture decision with genuine risk of getting a precedent wrong that the standing rule exists for, and it's the one idea in this batch that actually meets that bar.

Why this is architectural, not incremental: `BHC_Progress`'s entire data model — `step_status`, `mark_step_complete`, `completed_steps`, `first_incomplete_lesson`, `course_percent` — is built on the assumption that "next" is `step_index + 1` and that a lesson's total step count is a fixed, known integer. Every one of those methods would need to change shape, not just gain a parameter, to support "the next step depends on which answer you picked":

- **"What's next" stops being arithmetic and becomes a lookup.** A quiz step (or any step) needs a way to say "if the student picked answer B, go to step/lesson X instead of the next index" — that's a rule table (`from_step`, `condition`, `to_step`) that doesn't exist anywhere in the schema today, and every call site that currently does `$step_index + 1` needs to become "ask the rule table."
- **`course_percent()` becomes ill-defined.** Percent-complete assumes a fixed-size linear course. A branching course has no single "100%" — a student who took path A completed a different, non-comparable set of steps than one who took path B. This directly affects the catalog card's progress bar (just shipped in the last pass) and the "Continue"/"Review" CTA logic (`render_continue_cta()`) — both need a real answer to "percent of what?" before branching content can safely coexist with either.
- **Drip scheduling (`BHC_Gate::lesson_is_open`/`drip_notice`) currently schedules by lesson POSITION in a fixed order.** A branching course doesn't have a fixed position for a lesson that's only reachable via one of several paths — drip-by-calendar-day and branch-by-answer are two different axes that haven't been reasoned about together at all.
- **Authoring UX**: `BH_Studio`'s canvas is presented and rendered as a strictly linear top-to-bottom tree today. Branching needs either a genuinely different authoring view (closer to the mind-map tool in Section 4 than to the current linear canvas) or a bolted-on "if this answer, jump to..." control on top of the linear view that risks being confusing to author correctly — this is itself a design question, not just a backend one.

None of this means branching is a bad idea — it's a legitimate, high-value LMS feature (Duolingo's whole skill-tree/placement-test model runs on exactly this). It means the design pass needs to answer, before any code: whether progress tracking moves to a graph model outright or keeps a linear "position" concept with branch points as a bounded exception; how percent-complete and drip-scheduling get redefined; whether existing linear courses need any migration at all (they shouldn't, if the design is additive — but that needs to be a deliberate design constraint, not an assumption); and whether authoring branching logic is a Studio-canvas extension or leans on the mind-map tool from Section 4 as its actual authoring surface. That last question is worth resolving jointly with Section 4's own scoping rather than as two separate passes that might contradict each other.

## 4. Graphic mind-map authoring tool

**Roadmap-only this phase — not scoped for a build pass yet**, same treatment `ROADMAP-platform-evolution.md` gives Section 7 (social/marketing): naming it honestly rather than pretending it's a near-term feature.

This is closer to a new authoring surface than a feature, for reasons parallel to why `ROADMAP-platform-evolution.md` Section 3 already treats the general block-composition canvas as its own infrastructure project: it needs its own visual canvas (node/edge graph rendering — a real JS graph library, not an extension of anything currently in this codebase), its own data model (nodes and edges as first-class objects, not a repurposing of `BH_Content`'s nested-tree shape, which is a strictly hierarchical parent/child structure and doesn't natively represent a graph with cross-links), and a real, currently-unanswered design question about what a node actually IS — a whole lesson, a single concept, a step, a course? — before any UI can be designed around it.

The genuinely compelling reason to take this seriously rather than dismiss it: if Section 3's branching-path work ends up needing a non-linear authoring view anyway (as flagged above), the mind-map tool may not be a separate initiative at all — it could BE the branching-path authoring surface (a node-per-step/lesson graph where edges encode the "if this answer, go here" logic branching needs, visualized instead of hidden in a rule table). That reframes this from "a fourth, disconnected big idea" into "the likely authoring UI for idea #3," which changes the sequencing story considerably (see Section 5 below) and is worth deciding explicitly before either gets a real design pass, not discovered after both have been separately half-built.

Not proposing this gets built now. Flagging it as *possibly the same project as branching's authoring problem*, which is itself useful scoping information.

## 5. Suggested sequencing (dependency-driven, not ambition-driven)

1. **Interactive video overlays** (Section 1) — buildable now, independent of everything else here, extends an existing step type. No prerequisite work.
2. **Authoring tool maturation** (Section 2) — ongoing, extends `BH_Content`/`BH_Studio` incrementally; interactive video's annotation UI is itself a natural addition to this track once built.
3. **Branching + mind-map, jointly scoped, not built** — before either gets a line of code, run one Opus/Plan-mode pass that answers both at once: does branching need a graph authoring surface, and if so, is the mind-map tool that surface or a separate thing? Answering these together avoids building a rule-table-and-form UI for branching now and a disconnected graph canvas for mind-mapping later, only to discover they should have been the same system. This pass should also produce the actual data-model decision for `BHC_Progress` (linear-with-exceptions vs. real graph model) — that decision gates any implementation work on either idea.
4. Only after that joint design pass exists: branching's backend (progress/drip/percent-complete rework) and the mind-map/graph authoring UI can be scoped and sequenced against each other for real, informed by whichever direction the design pass actually recommends.

Everything in Sections 1 and 2 can proceed in parallel with the Section 3/4 design pass — neither touches `BHC_Progress`'s core model, so there's no risk of the two tracks colliding.
