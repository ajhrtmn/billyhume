# Own Ur Shit ecosystem — VISION

This replaces the lost VISION.md. It's meant to be read cold, months from now, by someone (possibly you) who's forgotten the details — a comprehensive context dump, not a changelog. Where something has a narrower, more detailed roadmap doc already (safety/metrics, feedback-and-courses), this doc summarizes and points there rather than repeating it in full.

## Mission — read this before anything else

Own Ur Shit is not a product or a startup pitch. It's **digital civil service infrastructure** — a public utility. The complexity of modern life requires tools for reaching an audience, running commerce, building community, and learning, and right now those tools are rented from a handful of big tech platforms who own the relationship, the data, and the terms. That's not acceptable as a permanent default. This ecosystem exists so someone can opt out of that entirely: stand up their own platform, own their audience and data outright, pay only for hosting, and never be one algorithm change or account suspension away from losing everything they built. This should be a right, freely available to anyone willing to self-host — not a subscription, not a walled garden. Own yourself digitally, on infrastructure you control.

**The stance behind that, stated plainly (AJ, 2026-07-17):** the platforms this replaces don't just charge money — they extract value from people who created it and give nothing structural back, and they do it by making themselves the unavoidable middleman between a creator and the audience that creator built. That's the thing being opposed here, named directly rather than softened into generic "platform risk" language: rent-seeking intermediaries who monetize a relationship they didn't create and can revoke at will. Own Ur Shit is the alternative — infrastructure a person owns outright, not a service they rent access to.

A few concrete commitments follow from that stance, worth stating as durable rules rather than aspirations:

- **No ads, ever — not as a business model, not as an opt-in monetization tool a site owner could enable for themselves, not anywhere in this codebase.** An artist can charge for their own work however they like (tiers, purchases, tips, pay-per-play — see `bh-monetization-woo`), because that's them setting terms with their own audience directly. An ad network is different in kind, not degree: it's a third party inserted between the artist and the audience specifically to extract value neither of them asked for. This ecosystem's own philosophy doesn't allow it, even though the code is open and an owner could technically bolt one on — the same way a constitution constrains what a government can do even though it's just paper. This is the equivalent clause for this platform.
- **Agency and choice over algorithmic curation.** Where this ecosystem touches content discovery/aggregation (the catalog, the registry, a future federated index), the default posture is: the person publishing decides how their own work is organized and surfaced, not a ranking algorithm optimizing for engagement time. Tumblr/MySpace-era personal curation — a page that looks like the person who owns it built it, not a template everyone's feed converges toward — is the reference feel, not a modern algorithmic feed.
- **Stability over churn.** This isn't software that needs a constant stream of feature updates to justify a subscription, because there's no subscription to justify. The bar for adding something new is closer to a constitutional amendment than a sprint backlog item: does this serve a genuinely fundamental, durable need (the basic digital-citizenship toolkit — commerce, community, learning, communication, file ownership — has been reasonably well understood for 20+ years of consumer computing), not "what would keep people engaged this quarter." Add features because they're needed, not because shipping constantly is itself the goal.
- **Federated, not centralized — including for social/discovery.** Where this ecosystem builds anything social-graph-shaped (bh-registry's artist directory today; a future federated index of self-hosted instances), the model is ActivityPub-style federation — many independently-owned instances that can discover and interoperate with each other — never one central server everyone has to register with. Each person's instance is theirs; federation is how instances find each other without requiring a company in the middle to run the directory. This is the same "protocol, not platform" instinct already applied to CalDAV/CardDAV/IMAP for the personal-productivity pillars below, extended to the social layer.

None of this is abstract ideology layered on top of an otherwise-ordinary product — it's the actual design constraint. A feature proposal that requires a central server this ecosystem's own users don't control, or that works by inserting a paid intermediary between an artist and their audience, fails review here regardless of how good the feature otherwise is. See "No vendor lock-in" and the self-hosted-always rules directly below for how this already shows up in day-to-day architecture decisions — the stance above is the reason those rules exist, not a separate, later addition to them.

**Every architectural decision has to serve that.** Concretely, in order of how often this actually comes up in practice:

- **Self-hosted, always.** Nothing in this ecosystem should require a server this ecosystem's own owner doesn't control. WP-Cron + MySQL on ordinary shared hosting (Bluehost-class, not a managed platform with special infrastructure) is the deliberate baseline every piece of shared plumbing (`OUS_Jobs`, `OUS_DebugLog`, `BH_Event`) is built against — see the event-tracking architecture pass for the most recent concrete example of this constraint actually shaping a design decision (no Redis/Memcached/external queue assumed, ever).
- **No vendor lock-in.** A feature should never be built in a way that only works if a specific third-party service stays alive, stays priced the way it is today, or keeps its API stable. Where a real external standard exists (CalDAV/CardDAV, IMAP/SMTP, OpenAPI), build against the STANDARD, not a specific vendor's implementation of it.
- **No quiet dependency on a paid third-party service where an owned equivalent will do** — analytics, CRM, LMS, email delivery, whatever. This ecosystem builds the owned equivalent itself (bh-crm instead of a CRM SaaS, `BH_Event`/`bhcore_events` instead of GA/Segment/Amplitude, `wp_mail()`-based notifications instead of a transactional-email API) specifically so the artist never has to trust a company that could raise prices, get acquired, or shut down.
- **If a design choice reintroduces dependence on big tech infrastructure — even indirectly, even for something small like a tracking pixel or an email API — that's a real finding, not a nitpick, and it needs to be named explicitly wherever it comes up.** This is now a standing review question for every new feature, the same weight already given to the `class_exists()`-guarded-peer-optionality rule below. See the event-tracking architecture plan's own "self-hosted posture flags" section for what this review actually looks like applied to a real design.

This framing doesn't change what gets built next so much as it changes HOW every future addition gets evaluated — a feature that's technically impressive but only works via a rented API is a worse outcome here than a plainer one that keeps working forever on a $10/month hosting account.

## What this actually is, today

One coherent product: an independent musician's business-in-a-box, built as a WordPress plugin ecosystem. One required core (`own-ur-shit`) plus genuine peer feature plugins that each depend ONLY on the core, never on each other:

- **bh-streaming** — personal streaming library, aggregator, Jam shared-listening.
- **bh-contest** — music contest voting/reveal/results.
- **bh-crm** — a person list/activity view built on shared identity.
- **bh-registry** — a decentralized, anonymous (no account required) artist-link directory.
- **bh-monetization-woo** — supporter tiers, purchases, tips, pay-per-play, all via WooCommerce, plus the generic `BHM_Gate` paywall any content type can use.
- **bh-courses** — an LMS: courses of multistep lessons (text/image/video/quiz), progress tracking, drip scheduling, optional tier-gating.
- **BH Feedback** — roadmapped, not built: paid song-feedback tiers (quick take / detailed / timestamped audio annotations).

The architectural rule that makes all of this work without turning into a tangled mess: **a peer plugin depends only on the core's presence, and treats every OTHER peer plugin as entirely optional**, checked via `class_exists()` at runtime (never at file-parse time) with a working fallback when the other plugin isn't there. bh-streaming doesn't know bh-monetization-woo exists except through a documented filter contract. bh-courses doesn't know bh-streaming exists at all. This is the single most important thing to preserve if this ecosystem keeps growing — it's what makes "install just the two plugins you actually want" a real, working claim rather than marketing copy.

## Current standing priority: harden core before adding anything new

Stated explicitly by the project owner and meant to override "what's next" until it's actually true: **core (`own-ur-shit`), bh-contest, the style system, bh-crm, and Debug Tools must be air-tight before any new feature work happens.** "Air-tight" means, concretely:

- **Jobs, errors/logs, testing, and the queue system are the load-bearing infrastructure everything else depends on** — bugs or gaps here don't stay local, they silently corrupt whatever's built on top (see the object-cache/transient saga earlier in this doc's own history: one unreliable primitive produced three unrelated-looking bugs before the shared root cause was found). Treat any fragility discovered in `OUS_Jobs`, `OUS_DebugLog`, `OUS_TestRunner`, or `OUS_ReliableStore` as higher priority than any feature request, always.
- **Abstractions/interfaces (`BH_Content`, `BH_Commerce`, `BHM_Gate`, `BHR_Links`, etc.) need their contracts kept clean and stable, not just their current call sites working.** This is the actual mechanism that keeps token/context cost manageable on a codebase this size going forward — a well-kept interface boundary means a future session can work inside `bh-monetization-woo` without ever needing to load `bh-courses`' internals into context, and vice versa. Every leaky abstraction found should be named and fixed here, not patched around at the call site.
- **API Docs and this codebase-learning documentation are themselves core deliverables now, not a nice-to-have.** The goal (see the new Docs-tab work below) is one browsable, in-admin reference compiled FROM the live route table and this walkthrough/roadmap material — not two things that can drift apart.
- **Coding-standard consistency matters even in isolated, single-file changes.** A change made deep inside one plugin should still read like it belongs to the same codebase as everything else — same logging conventions, same `class_exists()` guard shape, same doc-comment style — specifically because most future sessions will only have ONE file's worth of context loaded at a time; consistency is what lets a narrow diff be trusted without re-reading the whole surrounding system.
- **Prioritize breadth-first stability over depth-first features:** finish hardening one system fully (with real Debug Tools visibility and Test Runner coverage) before moving to the next, rather than lightly touching many systems in parallel. Each piece needs to hold its own integrity AND its API contract with its neighbors, since — per the peer-optionality rule above — everything here is built to interlock, not stand alone.
- **Before building new dev/debug tooling, check for a viable open-source option first** — something with no build step, that runs on ordinary shared hosting, matching this ecosystem's own no-vendor-lock-in / self-hosted-always posture (see Mission above). Hand-rolling is the fallback, not the default.
- **Standing suspicion to actively rule out, not just patch around:** given how many of this ecosystem's real bugs (Portal's rewrite rule, the API Docs "not allowed" error, the Live Console dropdown) have turned out to be environment/load-order/caching issues rather than pure logic bugs, any future "this feature that should work doesn't" report should specifically consider third-party plugin/theme conflicts and WordPress's own hook-timing/caching behavior as prime suspects before assuming the bug is in this ecosystem's own logic alone.
- **New dev/admin-only pages default to a Debug Tools SECTION, not a standalone `add_submenu_page()` page.** Concrete incident this responds to: `OUS_ApiDocs` and `OUS_CodebaseDocs` were registered as ordinary standalone admin pages (`add_menu_page`/`add_submenu_page` — one of the simplest, most universally-used WordPress APIs there is, used correctly the same way thousands of other plugins use it) and were consistently denied access on THIS install ("Sorry, you are not allowed to access this page"), five diagnostic passes deep, even with registration and `current_user_can('manage_options')` both directly confirmed successful via logging on the exact failing request. **Query Monitor (installed temporarily, capability-checks panel enabled via `define('QM_ENABLE_CAPS_PANEL', true)` in wp-config.php) finally isolated the real mechanism**, something no amount of custom logging from inside this codebase could have shown: `get_current_screen()`/`$hook_suffix` on the failing request resolved to `toplevel_page_ous-debug` — the PARENT page's own hook — not `admin_page_ous-codebase-docs`, the correctly-registered submenu hook `add_submenu_page()` itself had already confirmed returning. That means WordPress's own internal page-hook resolution (`get_plugin_page_hook()`, the function wp-admin's `admin.php` uses to match a `?page=` slug back to its registered callback) was failing to find the submenu's hook and silently falling back to the parent's screen context before denying access — a genuine, narrow WordPress-core lookup failure, NOT a caching issue and NOT a capability issue, the two things this session spent the most effort chasing first. The exact reason `get_plugin_page_hook()` fails specifically for these two pages (and not for `ous-debug`'s own auto-relabeled index submenu, registered under the same parent with the same pattern) is still unconfirmed, but the FAILURE MECHANISM itself is now correctly identified, which is what actually matters for deciding how to build the next page. Both pages were fixed by rendering their real content as sections on `ous-debug` instead (the `ous_debug_tools` filter, same pattern as Job Queue/Portal/every other working section) — that page's own hook resolves correctly every time, which sidesteps this failure mode entirely rather than needing to fully explain it. Treat a brand-new standalone admin page slug as a real, demonstrated risk on this specific install until this is more fully understood — Query Monitor stays a good diagnostic tool to reach for first if it happens again, rather than re-deriving log-based diagnostics from scratch.

This section is the active lens for prioritization until the project owner says otherwise — read it before picking up any new work in this repo.

## The core's shared services (own-ur-shit)

Everything below is provided ONCE, centrally, so no feature plugin reinvents it. This is genuinely the most reusable part of the whole ecosystem — read this section before building anything new.

- **Identity** (`BHI_*`) — registration/login/session, email verification, public profile pages.
- **Style** (`BHY_*`) — shared design tokens (colors/typography/spacing) plus a live-previewed Style Gallery any plugin registers a "surface" into via `bhy_style_surfaces`.
- **Reports/moderation** (`BHI_Reports`, table `bhi_reports`) — one shared abuse-report queue any plugin's "Report" button writes into.
- **Dashboard/registry** (`OUS_Dashboard`, `OUS_Registry`) — one install/activate/status dashboard; a new plugin joins it either via the `ous_registered_plugins` filter (rich card) or a zero-code `Ecosystem: Own Ur Shit` plugin header (auto-discovered, minimal card). Also the one place a THIRD-PARTY WordPress.org plugin (WooCommerce, Advanced Media Offloader) gets a one-click install card via `wporg_slug`.
- **Debug Tools** (`OUS_Debug`) — one shared, production-locked (`wp_get_environment_type()`-gated) page where every plugin registers a seed/reset section via `ous_debug_tools`. As of the QA/refactor pass below, there's now also an `OUS_Debug::register($key, $label, $render, $handle, $reset)` convenience wrapper around that same filter — every existing registrant (bh-contest, bh-streaming, bh-courses, bh-registry, bh-crm, bh-monetization-woo) still calls `add_filter('ous_debug_tools', ...)` directly and was deliberately left alone rather than retrofitted; the wrapper exists for whatever registers next.
- **Console & Logs** (`OUS_DebugLog`) — *shipped this pass.* The "no PHP CLI on this machine, no tailable log file" problem's actual answer: one DB-backed aggregate log (table `bhcore_debug_log`), capturing PHP fatals (via a shutdown-function safety net), WordPress's own `doing_it_wrong`/deprecated-function notices, and admin-only front/back-end JS errors (a tiny `window.onerror` → `sendBeacon` capture, enqueued only for logged-in `manage_options` users so it never adds JS payload to anonymous traffic), PLUS anything any plugin explicitly logs via `OUS_DebugLog::log($level, $message, $context, $source)`. Rendered as a filterable, colored table right on the Debug Tools page — same shared-page pattern as everything else. Opportunistically self-trims to the last 1000 rows on write, no separate cron needed.
- **Test Runner** (`OUS_TestRunner`) — *shipped this pass.* The other half of "no CLI needed": any plugin registers a suite via `bhcore_test_suites` (`['label' => ..., 'callback' => callable]`), where the callback returns a plain array of `['name' => ..., 'pass' => bool, 'message' => ...]` rows — no PHPUnit, no Composer, no separate dev environment, because the deployed WordPress site's own PHP IS a real PHP environment, it just lacks a CLI/test-runner binary on it. A "Run all tests" button on the Debug Tools page runs every registered suite live, right there, and renders pass/fail per case. **Real, working examples to copy from** (each ported from an existing PHPUnit suite in that plugin's own `tests/` directory, which stays in place for real CI later):
  - `own-ur-shit/includes/class-core-test-suite.php` — the RFC 6238 TOTP vectors + base32 codec tests, using `ReflectionMethod` to reach `BHI_TwoFactor`'s private methods directly, same as the PHPUnit version.
  - `bh-courses/includes/class-test-suite.php` — the quiz-scoring and step-sanitization edge cases (rounding, off-by-one threshold, clamping out-of-range indices, empty-quiz division-by-zero guard).
  - The PHPUnit versions of these same suites remain the ones to extend if this ecosystem ever gets a real CI pipeline with an actual PHP/Composer environment — the Test Runner versions are the "right now, no infra" complement, not a replacement.
- **API Docs** (`OUS_ApiDocs`) — *shipped this pass.* Turns whatever REST routes any plugin has ALREADY registered (`BHI_Auth`, `BHI_Reports`, bh-streaming's feed export, etc.) into a real OpenAPI 3.0 spec, generated live from WordPress's own route table (`rest_get_server()->get_routes()`) rather than hand-maintained separately and inevitably drifting from the actual code. Available two ways: the raw spec at `GET /wp-json/ous/v1/openapi.json` (importable into Postman/Insomnia/real Swagger UI), and a plain in-admin viewer at Own Ur Shit → API Docs with zero external JS/CDN dependency (consistent with this ecosystem's no-external-runtime-dependency convention — see the notification bell's own admin-bar-handle fix for the same instinct applied earlier). Only this ecosystem's own route namespaces are included, so WordPress core's huge built-in `wp/v2/*` table doesn't drown out the actual surface being documented.
- **Notifications** (`OUS_Notifications`) — *shipped this pass.* One call, `notify($user_id, $type, $title, $body, $url, $source, $email = true)`, writes to a shared in-app inbox (admin-bar bell + `[bh_notifications]` shortcode) and optionally queues an email through the job queue. **Real, working examples to copy from** (read these before adding a new one):
  - `bh-courses/includes/class-crm-integration.php` — notifies a student when they finish a course. The prototypical "good news" example.
  - `bh-monetization-woo/includes/class-products.php::on_order_reversed()` — notifies a customer their access/wallet credit was just reversed by a refund. The prototypical "transparency, not just good news" example — notifications aren't only for celebrations.
  - `bh-monetization-woo/includes/class-products.php::grant_entitlement()` — notifies on ANY entitlement grant (tier, purchase, subscription) from one single low-level choke point rather than duplicating the call in every higher-level caller. Also the one example passing `$email = false` (in-app only, since WooCommerce's own order-confirmation email already covers this) — the pattern for "don't double up on email channels that already exist elsewhere."
  - Note bh-registry was deliberately considered and rejected for a notification integration: registry submissions aren't tied to a WordPress account at all (anonymous, contact_email only), so there's no `$user_id` to notify — a good example of checking the data model actually fits before forcing an integration just for coverage's sake.
- **Jobs/queue** (`OUS_Jobs`) — *shipped this pass.* A WP-Cron-driven async queue (no external broker assumed), `enqueue($hook, $args, $delay)` / `register($hook, $callback)`, retry with exponential backoff, visible on the Debug Tools page (pending/running/done/failed counts, a "run now" button, recent failures). **Real, working examples to copy from:**
  - `bh-registry/includes/class-verification.php::recheck_all()` — the first one built: each link's daily re-verification becomes its own queued job instead of 50 running inline in one cron tick. Read this one first; it's the reference implementation the pattern below copies.
  - `bh-streaming/includes/class-feeds.php::sync_all()` — the second one: each external feed's twice-daily sync becomes its own job, so one slow/hanging feed can't delay every other feed source behind it. Written to look almost identical to the registry version on purpose — that repetition IS the point; a job-queue integration should be boringly predictable to add, not creatively different every time.
  - The pattern both follow: keep the original all-inline loop as the `else` branch (graceful degradation if `OUS_Jobs` isn't there), enqueue one job per unit of independent work in the `if (class_exists('OUS_Jobs'))` branch, and register the per-unit handler in the consuming plugin's own bootstrap with the same guard.
- **Optional 2FA** (`BHI_TwoFactor`) — *shipped this pass.* Off by default (site holder's call, Own Ur Shit → Security), per-account opt-in even when the site-wide switch is on, standard TOTP, enforced identically on wp-login.php and the REST login endpoint via one shared hook in WordPress's own `authenticate` filter chain (not two separate implementations).
- **Role scaffolding** (`OUS_Roles`) — *shipped this pass.* A `bhcore_manage_students` capability (bh-courses' Student Progress page now checks this instead of generic `edit_posts`) and a scaffolded-but-unused-yet `bhcore_review_submissions` for BH Feedback's future reviewer queue. Capabilities only — no role-assignment UI yet (see gaps below). Note the bug this caught during review: capability grants MUST also run on `init` (idempotently), not only on the WordPress activation hook — a file-replace deploy of an already-active plugin never fires that hook, a lesson this ecosystem's own DB-migration classes already knew and this one initially missed.

- **Events/identity** (`BH_Event`, `BH_Identity`) — *real and built, see EVENT-TRACKING-ARCHITECTURE-PLAN.md for the full design.* The self-hosted answer to "we're renting GA/Meta Pixel/Segment for something this ecosystem should own outright": one namespaced, versioned, schema-registered event envelope (`register_event_type($type, $v, $schema)`, `emit()`), ingested through `OUS_Jobs` (never a parallel mechanism — a play/vote beacon must never block the visitor's request on a DB write), stored in one core table (`bhcore_events`), joinable on one identity via `BH_Identity`'s anonymous `client_id` → `user_id` stitching (so pre-signup activity isn't lost the moment someone registers). Files: `own-ur-shit/includes/class-event.php` (`BH_Event`), `own-ur-shit/includes/class-identity.php` (`BH_Identity`); `bhcore_events` added to `class-identity-activator.php` (`DB_VERSION` 1.7→1.8). Real, wired consumers: bh-streaming (`BHS_Stats::record_play()`/`record_skip()` emit `bhs/play`/`bhs/skip`, replacing a synchronous per-play DB write with a queued one, BHS_VER 0.5.2), bh-contest (vote add/remove in `class-api.php` emit `bh/vote`, fired after — never inside — the vote's own transactional tally logic, BH_VER 3.1.5), bh-courses (`enroll_if_needed()`/`mark_step_complete()`/`maybe_fire_course_completed()` emit `bhc/enroll`/`bhc/step_completed`/`bhc/course_completed`, BHC_VER 0.4.4). **bh-crm is NOT wired yet** — it only exposes the `bh_crm_activity_summary` filter contract; no code currently reads `bhcore_events` into it, so a contact's Activity section does not yet include pre-signup event history. That's the next real consumer to build, not a rounding error. This pass also surfaced and fixed several genuinely synchronous, timeout-risking `wp_mail()` call sites in bh-contest (a mail-per-winner loop, a submission-confirmation send inside a file-upload request) that had nothing to do with tracking per se — see the architecture plan's own call-site audit.

**The standing instruction, restated plainly for whoever builds the next thing:** before adding a new cross-cutting concern, check whether it's actually a Notifications problem, a Jobs problem, a Roles problem, or an Events problem first — these four exist now specifically so the next plugin doesn't reinvent them.

## "It just works": guided setup as a design principle, not a feature (forever context)

Stated explicitly (AJ, 2026-07-17), and meant to sit alongside the design-system conventions below as an equally load-bearing rule, not a nice-to-have: **owning your infrastructure only beats renting it if the ownership part doesn't feel like a chore.** A self-hosted alternative that requires reading API documentation and hand-copying credentials between two different dashboards to get basic value (fast image delivery, reliable email, a CDN in front of media) will lose to the rented SaaS every time, even though the rented version is worse for the person long-term — because in the moment, the rented version just has a "Connect" button. This ecosystem has to beat that specific experience, not just beat the rented service's price or terms. Apple's "it just works" and TurboTax's guided-interview UX (ask one plain-language question at a time, translate the answer into the complex thing underneath, never show the raw complexity unless someone asks to see it) are the two named reference points — not because either company shares this project's politics, but because both solved exactly this problem: making a genuinely complicated backend (a filesystem, a tax return) disappear behind an interface a non-expert trusts on the first try.

**Concretely, what this means for any feature that wraps a third-party service or a genuinely technical integration** (a CDN/object-storage provider, a payment processor, a mail server, a future CalDAV/CardDAV account, anything with API keys/regions/buckets/DNS involved):

- **Default to a step-by-step wizard, not a settings page**, whenever setup involves more than "toggle this on." A settings page with 15 fields dumped in one screen is correct for someone who already knows what every field means; it's a wall for someone who doesn't, and this ecosystem's actual target user is very often the second person, not the first. A wizard asks one plain-language question per screen ("Where do you want your media files stored?" not "S3 Bucket Region"), shows/hides only the fields relevant to the answer just given, and explains WHY a step matters before asking for the technical value it needs.
- **Never require reading the third-party provider's own docs to finish setup.** Where a step needs a credential from an external dashboard (a Cloudflare API token, an Azure storage key), the wizard step should link directly to the exact page on that provider's dashboard where the value lives, with a screenshot or precise "click here, copy this" instruction — not "get your API key from Cloudflare" and leave the person to find it themselves.
- **Validate and confirm in real time, not just accept-and-hope.** A wizard step that collects a credential should test it immediately (a real API call, not just a format check) and say clearly "connected" or "that didn't work, here's why" before letting someone move to the next step — never let someone finish a whole wizard only to discover step 2 was wrong.
- **The complex, correct settings page still exists underneath** — a wizard is a guided PATH to the same configuration an expert could set directly, never a separate, dumbed-down feature set. Someone who already knows exactly what they want should be able to skip straight to the raw settings; the wizard is the on-ramp, not a wall between a non-expert and the real feature.
- **This applies ecosystem-wide, not just to media/CDN** — payment/commerce setup, a future email/calendar account connection, anything from the "big-vision pillars" sections below that touches a real external system, should get this same treatment as it's built, not bolted on after the fact. See `ROADMAP-guided-setup-wizards.md` for the concrete framework and the first real implementation (media/CDN offload), meant to become the reusable pattern every future integration wizard copies, the same way `OUS_Jobs`/`OUS_Notifications` became the reusable pattern for their own concern.

## Design system & style conventions (forever context)

Stated preference, worth keeping visible rather than re-litigating each time a UI decision comes up: **Apple/Google/GitHub as the reference design systems** (Apple HIG, Material Design, GitHub Primer), and **Storybook UI's own UX/GUI as a reference for how a component-gallery TOOL itself should feel** — not a library dependency, a taste reference. `BHY_UI`'s Style Gallery already independently arrived at a "Storybook-patterned live preview gallery" shape before this preference was written down explicitly here; that's confirmation the instinct was already right, not a new direction.

The standing rule this implies, and the one that actually got tested this pass: **default to plain, unmodified WordPress admin styling everywhere; deviate only where there's a genuine, specific UX win, and make that deviation a shared, reusable, documented piece of `BHY_UI` — never a one-off inline style.** `.bhy-table-wrap` is the concrete precedent: WordPress's own wide admin tables (`wp-list-table`/`widefat`) don't scroll or adapt on a narrow viewport, which is a real, demonstrated problem (a per-lesson-column student progress table, a multi-column stats table) — not a cosmetic preference. The fix is one shared class (sticky header, horizontal scroll, denser padding under a CSS container query), applied consistently across every ALREADY-EXISTING wide table in the ecosystem during this pass (own-ur-shit's reports table, bh-registry's admin table, bh-monetization-woo's CRM table, bh-streaming's four stats tables + CRM table, bh-courses' CRM detail + progress-admin tables — 11 tables across 6 files, plus bh-crm confirmed already doing this correctly on its own). Nothing else about admin styling was touched — no broad re-theming, no opinionated recoloring of stock WP chrome. The next person adding a wide table should reach for `.bhy-table-wrap` by default; the next person tempted to restyle something ELSE that isn't a genuinely wide/dense table should probably not.

### New features get Debug Tools + Test Runner coverage where it actually adds value (forever context)

Stated explicitly after this pass's own bug-fix-and-observability work, worth keeping visible so the INSTINCT doesn't get lost on a future pass: **default to adding Debug Tools/Test Runner coverage in the same pass a feature ships, but this is a judgment call, not a mechanical checkbox.** The bar is "does this actually let a person verify/track/diagnose something real," not "does every function technically have a test." A trivial getter, a pure passthrough, or a one-line wrapper around a WordPress core call doesn't need a dedicated suite entry just to say it has one — that's coverage theater, and it's exactly the kind of padding that makes the REAL coverage harder to find later. Two questions worth actually asking before adding either:

1. **Debug Tools: can this silently fail/misconfigure in a way a person would otherwise have to re-read source to diagnose?** If yes, log why (via `OUS_DebugLog`) and/or surface current state on Debug Tools — see `OUS_ApiDocs::add_menu()` now logging its own skip reason instead of silently returning, and `BH_Studio`/`BHI_Portal`'s registered-block-types/registered-panels tables. If a feature genuinely can't misconfigure silently (its failure mode is always loud — a PHP fatal, a visible empty state with an obvious cause), it doesn't need a dedicated Debug Tools section just for the sake of one.
2. **Test Runner: is there real logic here worth pinning down, or just plumbing?** Pure functions with actual edge cases (rounding, boundary conditions, coercion rules) are the highest-value target and the cheapest to write — see `BHM_Gate::calculate_downgrade_credit_cents()` (deliberately extracted out of its DB-touching caller specifically to be testable) and `OUS_StudioTestSuite`'s block validate()/render() round-trip. DB-backed logic awkward to unit-test in isolation is often better covered by a Debug Tools seed/reset action that exercises it end-to-end (see `BHM_Debug`'s tier/entitlement/wallet buttons) than by forcing a fixture-heavy "unit" test. Something with no real branching or edge cases — skip it, and don't feel obligated to manufacture an assertion just to pad a suite.

This matters more than usual for this specific codebase because no session working on it has ever had a live PHP/MySQL/WordPress execution environment (see `HANDOFF-PROMPT-v25.md`'s own Step 0). Where coverage IS warranted, it's the only in-repo mechanism letting a person with a real install verify a change actually works — but the goal is genuine signal a person can act on, never volume for its own sake.

### Smart error handling, tracing, and garbage collection (forever context)

Three habits worth being deliberate about on every new piece of state/logic, same "where it adds real value" bar as the section above — not a mandate to wrap every line in a try/catch:

- **Error handling should fail toward a clear, specific signal, not a silent no-op or a generic catch-all.** `OUS_DebugLog::log_exception()`/`capture_uncaught_exception()` exist so a `catch` block can record what actually broke (file/line/trace) instead of swallowing it — reach for those rather than an empty `catch (\Throwable $e) {}` unless the code is ALREADY in a context where logging itself could cascade (shutdown handlers, the logger's own internals — see `OUS_DebugLog::capture_fatal_on_shutdown()`'s own defensive catch for the one legitimate exception to this).
- **Tracing:** now that `OUS_DebugLog` v2 carries structured file/line/column and full stack traces (not just a message string), new code that logs an error should pass along real trace data where it's cheaply available (a caught `Throwable`, a JS `error.stack`) rather than a bare string — see `log_exception()`'s signature. Don't hand-roll a second tracing mechanism anywhere; this is the one shared path.
- **Garbage collection / bounded growth:** any new table or log that grows unboundedly needs a trim/expiry story from day one, not a "we'll deal with it later" — `OUS_DebugLog`'s own opportunistic `maybe_trim()` (self-caps at `MAX_ROWS` on ~1/50 writes, no separate cron) is the reference pattern: cheap, no new scheduled task, no unbounded table. `OUS_Jobs`' queue and `bhcore_notifications` are the other two existing examples worth copying from rather than reinventing. A table that's naturally bounded by real-world cardinality (one row per user, one row per tier) doesn't need this — the bar is specifically "can this grow forever from normal usage."

### WooCommerce integration — "feels like a true extension of both"

The concrete decision made this pass: `bh-monetization-woo`'s Supporter Tiers screen now nests under WooCommerce's OWN top-level admin menu (`show_in_menu => 'woocommerce'`) when WooCommerce is active, falling back to its own top-level menu when it isn't yet installed — see `class-tiers.php`. This mirrors exactly how real WooCommerce extensions (Subscriptions, Bookings) behave: a WooCommerce-savvy admin already expects money-adjacent settings to live under WooCommerce's own sidebar entry, not compete against it for space. The fallback matters because this plugin's whole premise needs WooCommerce eventually but must still be usable (or at least visible) during the brief window before it's installed.

Further idiomatic wins worth considering later, not yet built: a WooCommerce Settings **tab** (the way Subscriptions/Bookings add their own tab under WooCommerce → Settings) rather than a fully separate settings screen; matching WooCommerce's own product-data-panel tab conventions if/when tier products ever need custom fields exposed there; WC-style admin notices/badges for consistency. None of these are required — the menu-nesting fix is the one that mattered most for "doesn't feel bolted on."

## QA pass and DRY/SOLID refactor — what changed and why (this session)

Important context for reading everything below: this pass ran in an environment with **no PHP, no MySQL, no Docker, no root, and no outbound network access at all** — genuinely nothing to execute code with, confirmed by trying (apt-get fails on a permission-denied dpkg lock; every outbound curl, including to plain domains like google.com, hits a default-deny egress proxy). A previous attempt at this same environment left a `/tmp/phpinstall.log` behind showing an earlier identical failure. A remote-sandbox probe was also tried and came back with the same three independent walls (no root, no package manager, no egress). So despite the original brief asking for a real WordPress+MySQL install and live execution of every flow, that literally could not happen here — everything below is the best achievable **static** pass instead, and that distinction matters for how much to trust it.

What static analysis CAN still do well, and what it did:
- A purpose-built PHP-lexing checker (not the naive brace-counting script earlier sessions used, which had known false positives on regex character classes and CSS selectors) was written and run across all 100+ PHP files in the ecosystem — it correctly tokenizes single/double-quoted strings, heredoc/nowdoc, comments, and `<?php`/`?>` mode transitions before counting braces/parens/brackets, so it reliably catches the "unescaped apostrophe silently terminates a long single-quoted string" bug class this ecosystem has hit before. Result: zero unbalanced-brace/unterminated-string findings across the whole codebase as of this pass.
- Seven parallel deep-read passes (one per plugin, plus one for the core) cross-checked every `[self::class, 'method']`/cross-class call against its actual definition, every `class_exists()` guard's placement (call-time inside a hook vs. file-parse time), every nonce/capability check on state-changing actions, every `$wpdb->prepare()` call, and output-escaping coverage.

Real bugs found and fixed this pass (the kind that only surface at runtime, which is exactly why "reasoned through carefully but never executed" isn't the same as tested):
- **bh-streaming `class-player.php`** — the one unconditional `BHY_Style::inline_css()` call in the whole ecosystem with no `class_exists()` guard around it. Every `[bh_streaming]` shortcode page would have fataled if `own-ur-shit` (which defines `BHY_Style`) weren't loaded at that exact point — now guarded like every other cross-plugin touch.
- **bh-streaming `class-recommendations.php`** — the `/tracks/{id}/related` REST endpoint resolved audio URLs directly from postmeta instead of going through `BHS_API::track_payload()`, which meant it (a) shipped a real, playable URL for tracks a monetization plugin had locked — the one access-control mechanism this plugin has, bypassed — and (b) silently dropped every externally-aggregated/feed-imported track from recommendations entirely, since those store audio in `_bhs_external_audio_url` rather than `_bhs_audio_id`. Now routes through `track_payload()` like every other read path.
- **bh-courses `class-progress.php`** — `mark_step_complete()` passed `null` (for a plain text/image step's score/passed, which are legitimately NULL columns) through `$wpdb->prepare()`'s `%s`/`%d` placeholders, which have no NULL passthrough and cast `null` to `''`/`0`. Since `is_step_complete()` explicitly tests `=== null` to detect "non-quiz step, complete on write," this meant **every plain content step in every course permanently read as incomplete** — course-percent, resume-where-you-left-off, and completion detection were all silently broken. Fixed by building the SQL with a literal `NULL` when the value is actually null, instead of binding it through a placeholder.
- **bh-courses `class-progress.php`** — a quiz step that had already been passed could still be resubmitted and rescored, and a failing resubmission would overwrite the passing result (un-completing a step that had already counted toward course progress). Now short-circuits once passed.
- **bh-courses `class-gate.php`** — drip-content date comparisons mixed `current_time('timestamp')` (WordPress's site-timezone-adjusted "now") with raw `strtotime()` of stored date strings (PHP-default-timezone, not necessarily the site's). Fixed to resolve the admin-picked calendar date against `wp_timezone()` explicitly and compare enrollment timestamps against raw UTC `time()`, consistently.
- **bh-contest `class-api.php`** — the vote endpoint checked that a submission belonged to the right contest, but never checked `post_status === 'publish'` — a user who knew or guessed a pending/rejected submission's ID could burn one of their limited per-category votes on a track that could never appear in results.
- **own-ur-shit `class-public-profile.php`** — the avatar/banner upload handler had no capability check at all before calling `media_handle_upload()` (which performs none of its own); since `BHI_Auth::register()` creates plain subscriber accounts, this was a real "any self-registered low-privilege user can write into the media library" gap. The feature is explicitly meant to work for subscriber-level users, so the fix is a scoped, request-duration-only `upload_files` grant around just this call (via `user_has_cap`) rather than a hard block that would have broken the feature.
- **own-ur-shit `class-registry.php`** — `status()`/`version()` indexed the registry array directly instead of `isset()`-guarding like `OUS_ActivationManager` already does elsewhere; hardened to fail gracefully on an unknown key instead of an undefined-array-key warning.
- **bh-streaming `class-feeds.php`** — a feed-fetch network failure (timeout, DNS, host down) was indistinguishable from a genuinely malformed feed in the admin UI, both showing "didn't look like an open podcast/audio feed" — pointed admins at the wrong problem. Now tracked as a distinct `_bhs_feed_fetch_failed` state with its own message.

Encapsulation/abstraction work done alongside the bug fixes (the person running this session specifically asked that this get real attention, not just DRY-driven dedup):
- **`bh-monetization-woo`'s `class-products.php` was doing five genuinely distinct things in one class** (WooCommerce product sync, streaming-gate hooks, pay-per-play billing, refund/fraud-pattern detection, entitlement granting) — the single largest, most obviously overloaded file in the ecosystem. Refund/fraud-pattern detection (the most self-contained of the five) was extracted into a new `BHM_Fraud` class (`includes/class-fraud.php`), with its own private `fingerprint_for()` helper properly encapsulated inside it rather than exposed. `class-products.php` now just calls `BHM_Fraud::track_refund_pattern(...)`; `class-crm-integration.php`'s external read of `refund_count_recent()` was updated to call the new class. The other four responsibility groups (product sync, gating, billing, order/entitlement lifecycle) are more tightly coupled to each other (gating and billing both depend on `is_non_catalog_track()`, for instance) and were left as one class rather than force-split without a live environment to verify the split didn't break a call site.
- **`bh-registry` had no data-access layer at all for its own `bhr_links` table** — 14 call sites across 6 files each independently reached into `$wpdb->prefix . 'bhr_links'` and hand-wrote SQL, including three byte-identical "fetch one link by id" queries (`class-api.php`, `class-admin.php`, `class-verification.php`). A small `BHR_Links` class (`table()`, `find()`) now centralizes the table name and that duplicated lookup; the other, more varied queries (filtered lists, bulk deletes) were deliberately left where they are rather than forcing every access pattern through a heavier repository abstraction this plugin doesn't actually need yet.
- Broader encapsulation gaps that were **identified but deliberately not touched this pass**, flagged here so they're not lost: this whole ecosystem is written as static-method "utility bag" classes (idiomatic for a hook-driven WordPress plugin, not being proposed as wrong), but several plugins still repeat the same meta-key/table-name string literals across many files instead of going through one accessor (bh-monetization-woo's `bhm_entitlements` table name, cents-to-dollar-string formatting repeated ~10+ times, the `class_exists('WooCommerce')` guard repeated at the top of nearly every public method rather than centralized). None of these are bugs, and retrofitting all of them without a live WordPress install to verify each call site still behaves identically was judged higher-risk than it's worth in one static-only pass — a good scoped task for whoever next has a real test environment.

What this pass explicitly could NOT verify, because it requires actually running the code: plugin activation (fresh site or upgrading a site with pre-existing data, which is the whole point of the versioned migrations described elsewhere in this doc), every admin screen actually rendering without a fatal, the broadened screen-ID substring matching (`class-ui.php`) actually matching every real WordPress-generated screen ID it claims to, any front-end flow end to end (course enrollment/quiz-taking, contest voting, streaming playback, checkout), the Debug Tools seed/reset actions, the Test Runner's "Run all tests" button, both PHPUnit suites, and anything needing a live third-party dependency (a real payment gateway, a Cloudflare R2 bucket, a second WordPress instance for the registry's cross-instance verification, an actual reverse-proxy/CDN in front of the site to test the `REMOTE_ADDR`-based rate-limiting/fingerprinting caveats noted in `class-fraud.php` and `class-api.php`). Treat every fix above as "the reasoning is sound and the code now reads correctly," not as "confirmed working" — the next session with a real WordPress+MySQL environment should prioritize actually exercising all of this over another round of static reading.

## Near-term roadmap

### Rounding out what just shipped
- A real role-ASSIGNMENT admin screen (checkboxes per user, per registered capability) — right now granting `bhcore_manage_students` to a non-admin needs a one-off `add_cap()` call or the `bhcore_role_capabilities` filter, no UI.
- Per-notification-TYPE email preferences, not just the current all-or-nothing opt-out.
- A **front-end user bar** — distinct from WordPress's own wp-admin toolbar (which is admin-oriented, often hidden entirely on the front end, and easy for an ordinary fan/student account to never see). A slim, always-there bar for logged-in ordinary users specifically: notification bell (reusing `OUS_Notifications`), quick links contributed by any plugin via a filter (bh-courses' "continue learning," bh-streaming's "now playing," a plain account/logout), styled with the shared design tokens. The wp-admin toolbar solves this for admins; nothing currently solves it for the actual target user of most of these plugins — a fan, a student, a supporter — who may rarely if ever see wp-admin at all.
- BH Feedback (still just a plan — quick-take/detailed/timestamped-audio tiers, `BHM_Wallet`-backed one-time pricing, `bhcore_review_submissions` capability already scaffolded and waiting).
- BH Courses' remaining honest gaps: instructor notes (private, separate from student-facing content), discussion/Q&A per lesson, cohort/scheduled-release courses, instructor-graded (human-reviewed, not auto-scored) steps — the last one architecturally overlapping enough with BH Feedback's review queue that they should be designed together, not as two separate systems.

### Metrics — a real creator dashboard, not just per-plugin stats pages
Right now every plugin's numbers live in its own silo: bh-streaming has its own aggregate metrics dashboard (plays, skip-rate, geo/referrer — see the safety/metrics roadmap), bh-monetization-woo has wallet/entitlement ledgers, bh-courses has per-course student progress, bh-registry has verification status. None of it rolls up into one place an artist actually looks at to answer "how's my whole thing doing this month." The natural fix, matching a pattern already proven twice (bh-crm's `bh_crm_activity_summary`, and the general "any plugin contributes, zero central registration" shape `OUS_Notifications`/`OUS_Jobs` also use): a `bhcore_metrics_widgets` filter, one shared "Metrics" dashboard page in the core, each plugin contributing its own KPI card (plays this week, revenue this month, active students, verified artists, whatever it actually owns) with zero changes needed to the core when a new plugin wants a card. Genuinely worth building before too many more per-plugin metrics screens accumulate independently.

### Marketing tools — currently nothing exists here at all
Worth naming the actual shape of this gap rather than leaving it vague:
- **Email broadcast/campaigns** — "email everyone in tier X" or "email everyone who finished course Y" is exactly the kind of thing `OUS_Notifications`'s email channel is NOT built for (it's one-to-one, triggered by an event) — a real campaign tool is a different, segmented, admin-composed-and-sent concept, though it could reuse the same delivery/queue plumbing underneath.
- **Referral/affiliate tracking** — who sent a new supporter, was there a discount code, revenue attribution. Nothing like this exists; would likely live inside or alongside bh-monetization-woo given the money angle.
- **Link-in-bio / landing page tool** — the "one link for everything" page a lot of working musicians currently pay Linktree/Beacons for. A near-perfect fit for this ecosystem's own identity+style system; genuinely low-effort relative to value given the public-profile infrastructure already exists.
- **SEO basics** for the already-public-facing surfaces (registry browse pages, public profiles, course catalog) — meta tags, sitemaps, structured data. Nobody has looked at this yet.
- This entire category was flagged as explicitly important to keep visible in this roadmap — don't let it quietly drop off in favor of whatever feature is most top-of-mind next.

### Auction site — new addition to the roadmap
Not yet scoped in detail — flagged here so it isn't lost, with the real design work deferred to `ROADMAP-platform-evolution.md`'s storefront section once that's ready to be picked up. The shape worth sketching now: a time-boxed bidding mechanism (starting price, minimum increment, close time, optional reserve/buy-it-now) layered on `bh-monetization-woo`'s existing WooCommerce-backed commerce plumbing rather than a parallel payments stack — same "own the interface, don't reinvent what WooCommerce/`BH_Commerce` already does for checkout/fulfillment" posture the rest of this ecosystem already follows. Natural fits: one-of-a-kind merch, signed items, limited-run physical releases, a fan-funded "name your price with a floor" mechanic distinct from the existing flat-price tier/purchase model. Real open questions before this gets built for real: does a bid reservation need `BHM_Wallet`-style holds (so a bidder's payment method is authorized but not charged until they win), how does an auction closing interact with the existing refund/fraud-pattern detection in `BHM_Fraud`, and does this want its own post type or reuse the existing product/tier machinery with an "auction" pricing mode. Worth designing alongside the storefront layer in `ROADMAP-platform-evolution.md`, not as a standalone bolt-on, given both need the same product/commerce foundation.

### Flagged blind spots, still open
- **Tests exist now, but only where it counted most, and only two shapes of test.** TOTP/base32 and quiz-scoring/step-sanitization have real edge-case coverage (both as PHPUnit suites in each plugin's `tests/`, and as GUI-runnable suites via `OUS_TestRunner` — see above). Nothing that touches `$wpdb` directly (wallet debits, entitlement grants, refund reversal, the job queue's atomic claim) has a test yet — that's genuinely harder to test meaningfully without a real WordPress+DB integration-test environment, and was deliberately left honest rather than faked with a mocked `$wpdb` that would mostly test the mock.
- **Error visibility exists now for the touchpoints most likely to silently lose money or break trust** (a WooCommerce order-completion webhook, an external feed fetch, a registry link re-verification) via `OUS_DebugLog`, wrapped in try/catch that logs-and-continues rather than fatals mid-loop. Most OTHER touchpoints (AJAX handlers, admin form saves) still rely on WordPress's own default error behavior — worth another pass if a specific one turns out to matter in practice, rather than defensively wrapping everything up front.
- **No internationalization.** One file in the whole codebase uses `__()`. Fine for single-artist/English-only; a real gap for "creator shop" ambitions beyond that.
- **No site-wide search** spanning courses + tracks + artists + CRM notes.
- **No unified public profile** tying one person's streaming/courses/contest history together — currently siloed per-plugin despite sharing one identity system.

## The big-vision pillars, round two: enterprise tooling + personal/small-business finance

Added on top of the pillars above, same "separate product surface, same underlying hub-and-peer-plugins architecture if pursued" framing — and the same "own the interfaces, don't necessarily reimplement the hard 3rd-party pieces on day one" posture `ROADMAP-platform-evolution.md` Section 0 already lays out for WordPress/WooCommerce itself. The through-line connecting all of these to everything above: **remove third parties wherever realistically possible**, for both the business (the music-business vertical's own tooling) and personal life (email, calendar, accounting, taxes) — not as an ideological stance, but because every 3rd-party dependency named below is either actively being deprecated out from under its own users, gated behind enterprise pricing, or both, which is exactly the kind of instability this ecosystem's whole "own the interfaces" posture is meant to insulate against.

Current-state research (July 2026), same discipline as Section 7's social/marketing research — verify against live docs before scoping real work, not just this summary:

- **BI/reporting (replaces SQL Server Reporting Services / Report Builder)** — genuinely well-timed to target: Microsoft is actively sunsetting SSRS (SQL Server 2022 is confirmed as the last release), and its replacement, Power BI Paginated Reports, is not simply "the same free tool renamed" — as of 2026 it requires Power BI Premium Per User (~$24/user/month), a Microsoft Fabric capacity SKU (F64+), or self-hosted Power BI Report Server, none of which is a free tier the way classic Report Builder was. That's a real, current pain point for exactly the kind of small/mid-size shop that historically ran SSRS specifically because it didn't cost anything beyond a SQL Server license. A `BH_Reports`-style module (structured query → parameterized, paginated, printable report layout, scheduled email delivery) is a well-scoped, well-precedented feature shape — genuinely more tractable than the BI-dashboard space generally, since paginated/parameterized reporting is a narrower, better-understood problem than open-ended interactive analytics.
- **ERP/accounting (replaces NetSuite, and by extension QuickBooks Online/Xero for the smaller end of that same job)** — the honest framing here matters most: NetSuite (43,000+ companies, dominant mid-market cloud ERP per Oracle's own 2025 numbers) is a genuinely enormous, decades-deep product surface — general ledger, multi-entity/multi-subsidiary consolidation, inventory/order management, CRM, and more, all built for real accountants who need every one of those to be actually correct, not approximately correct. Realistic scope for THIS ecosystem is not "rebuild NetSuite" — it's the two things a solo creator or small business actually touches day to day: basic double-entry bookkeeping (income/expense categorization, invoicing, a chart of accounts) and light payroll/contractor payment tracking, both already adjacent to what `bh-monetization-woo`'s wallet/entitlement ledger and CRM already track on the revenue side. An interface-first posture applies here too — a `BH_Ledger` interface with a plain-MySQL-backed implementation now, QuickBooks Online/Xero as optional SYNC targets (both have modern, well-documented REST APIs, and QuickBooks in particular is built around the US CPA handoff workflow a lot of small businesses already depend on) rather than something this ecosystem needs to replace outright for anyone still working with an accountant.
- **Taxes (individual + small business)** — the most legally/regulatorily loaded item on this whole list, worth naming plainly rather than glossing over: IRS Direct File (the government's own free e-file tool) was discontinued for the 2026 filing season with no announced replacement, which if anything strengthens rather than weakens the case for an independent option — but actual e-filing requires IRS e-file provider authorization (a real, audited approval process, not just "write software that produces a 1040"), and correctness here has real financial and legal consequences for whoever uses it. Realistic first scope: a self-service ledger-to-tax-summary tool (categorized income/expenses → Schedule C-shaped summary a person hands to a real preparer or uses with existing Free File Fillable Forms) rather than an end-to-end authorized e-filer — the second is a multi-year regulatory undertaking on top of the engineering, not a feature sprint, the same honest distinction `ROADMAP-platform-evolution.md` Section 0 already draws for "replace WordPress entirely."
- **Email + calendar (replaces Gmail/Google Workspace, Outlook/Microsoft 365 for personal + small-business use)** — the standards-based path is real and well-trodden: CalDAV/CardDAV (the same open protocol Apple iCloud, Fastmail, and Proton Calendar all speak) has mature self-hosted server implementations (Radicale, Baïkal, Nextcloud's own CalDAV layer) worth building on rather than reinventing, and IMAP/SMTP are equally standard on the email side (Nodemailer-equivalent send, IMAP for receive/sync). The realistic build here is a `BH_Mail`-interface-backed (see `ROADMAP-platform-evolution.md` Section 2 item 4 — already scoped, just not built yet) webmail + calendar front end that SPEAKS these open protocols against either a self-hosted server this ecosystem also provides, or an existing IMAP/CalDAV-compatible account a person already has — genuinely more "assemble proven open standards behind our own UI" than "invent new infrastructure," which is the same reason `BH_Content` leaned on Gutenberg's existing block model instead of a from-scratch canvas.
- **The unifying architectural note:** every item above is a genuinely separate product surface from the music-business vertical (same as files/design/PM above), but ALL of them — reports, ledger entries, tax summaries, inbox/calendar items — are themselves naturally `BH_Content`-shaped structured/tabular data once that interface exists, and several (invoice line items, transaction categorization rules) are exactly the kind of "spreadsheet-like row/column editing beats drag-drop" case `ROADMAP-platform-evolution.md` Section 3 already flagged as a distinct UI mode worth building. Nothing here needs a new foundational abstraction invented from scratch — it needs the ones already being built (`BH_Content`, `BH_Commerce`, the still-unbuilt `BH_Mail`) applied to a wider set of domains than just the music business.

**Same decision this doc already asked, now with more pillars stacked onto the same fork:** is the music-business vertical the whole product with an ever-growing pile of "someday" stretch goals, or does "remove third parties wherever possible" mean this is meant to become a genuinely general personal + small-business operating layer, with the music-business plugins as just the first, most fully-built vertical? Worth deciding on purpose — the list of somedays is now long enough (files, design, PM, reporting, ERP/accounting, taxes, email, calendar) that treating all of them as equally "eventually" risks the decision getting made by accident rather than on purpose, same risk this doc already flagged once above.

## The big-vision pillars, round three: federated, ad-free social/discovery layer

Added alongside round two, same "separate product surface, same architecture if pursued" framing (AJ, 2026-07-17). Not entirely new — `bh-registry`'s README already scopes ActivityPub actor links and names "ActivityPub Follow/Accept federation" as a real, not-yet-built next step — this section makes that ambition explicit at the VISION level instead of leaving it implicit in one plugin's own doc, and states the philosophy driving it (see the Mission section's federated-not-centralized commitment above).

The shape: a person's Own Ur Shit instance (their `bh-streaming` catalog, their public profile, their courses) is discoverable and followable the same way a Mastodon/Pixelfed account is — via ActivityPub — without requiring anyone to register with a central Own Ur Shit-run server to find or follow them. `bh-registry` is the seed of the discovery half of this (a directory of instances that opted in); the federation half (actual Follow/Accept, activity delivery, a follower's own feed showing another instance's posts) is the larger, not-yet-scoped piece. Explicitly NOT a recommendation-algorithm feed — federation here means "make one person's own curation reachable by people who choose to follow it," not "build a ranking engine to compete with the platforms this replaces." Real prior art worth studying before scoping (Mastodon, Pixelfed, PeerTube, Funkwhale — bh-streaming's own README already references Funkwhale as a comparable) rather than inventing federation semantics from scratch; ActivityPub itself is the W3C standard doing the actual interoperability work, matching this ecosystem's existing "protocol, not platform" posture for CalDAV/IMAP.

**Federated metrics, not yet scoped — a hard constraint stated up front.** Once own-ur-shit's own creator dashboard (`OUS_Metrics`, built this pass — see `class-metrics.php`'s own docblock) exists on every instance, AJ raised the natural next step: aggregate metrics ACROSS distributed instances too, not just within one site. Genuinely useful — "how is the whole ecosystem doing," not just one artist's own numbers — but also the exact shape of thing this whole ecosystem's anti-middleman, no-vendor-lock-in stance exists to avoid if built carelessly: a central server quietly compiling everyone's real usage data is just a smaller-scale version of what Google Analytics/Meta already do. AJ's own follow-up, stated live and verbatim, is the design constraint any future work on this MUST satisfy before a single line of code: **"stripped of specific identities and a fair and transparent way of expressing the data."** Concretely, that means: anonymized/aggregate-only payloads leaving any instance (no user IDs, no raw per-visitor rows — same posture `OUS_Metrics` already takes locally), opt-in per instance (an instance owner decides whether to contribute at all, not a default-on telemetry pipe), and the aggregation method and resulting numbers need to be independently inspectable/auditable, not a black box only the aggregator can see inside. Not built yet — deliberately left unscoped rather than designed under time pressure, given how easily this specific feature could drift into the thing the rest of this document argues against.

## Platform-evolution interfaces — what shipped this pass (see ROADMAP-platform-evolution.md)

This pass ran in the same walled-off environment as the QA/refactor pass above (no PHP, no MySQL, no root, no outbound network — reconfirmed by testing, not assumed) — so everything below is again "the reasoning is sound and the code now reads correctly," not "confirmed working." Job 1 of this handoff (actually running QA-REPORT.md Section 5 against a real WordPress+MySQL install) could not happen here either, for the same reason; skipped explicitly rather than silently, per this handoff's own instructions. Job 2 — the three new interfaces, in the roadmap's own dependency order — is what got built:

- **`BH_Content`** (`own-ur-shit/includes/class-content.php`) — the content-block interface, shipped as the thin layer over Gutenberg's own block model the roadmap doc recommended as the first implementation. `register_block_type($type, $schema, $renderer)`, `validate()`, `render()`, and a `get()`/`save()` storage pair that uses a real post's `post_content` (via `parse_blocks()`/`serialize_blocks()`) for post-attached documents, and a new `bhcore_content` table (JSON blocks column, `bhi_db_version` bumped to 1.5) for everything else. Real, working consumer: **bh-courses' lesson-step authoring** (`bh-courses/includes/class-content-bridge.php`) — registers `bhc/text`, `bhc/image`, `bhc/video`, `bhc/quiz` block types, converts the existing `_bhc_steps` array to/from a `BH_Content` tree, and keeps both in sync on save so `BHC_Progress`/`BHC_Render`/`BHC_Gate` (all step-index-based) needed zero changes. A "rebuild BH_Content trees" button is registered on the shared Debug Tools page as the concrete, clickable way to exercise the migration today, ahead of any real visual authoring UI.
- **`BHI_Portal`** (`own-ur-shit/includes/class-portal.php`) — the account-area shell at a rewrite-owned `/account/` URL, built from panels contributed via the new `bhi_portal_panels` filter (same zero-central-registration shape as `ous_registered_plugins`/`ous_debug_tools`). Ships with the wp-admin exclusion rollout (`admin_init` redirect + `show_admin_bar` filter for `subscriber`/`customer` roles, exemptions via `bhi_portal_excluded_roles`) and exactly one real migrated panel — profile/identity, via `BHI_PublicProfile::render_portal_panel()` (a thin wrapper around the existing `render_edit_form()`; the standalone public profile page/shortcode is untouched, since that's a deliberately different, shareable surface). Every other panel named in the roadmap (subscription/wallet, course progress, contest submissions, notifications) is a documented next step, not built yet — matching the roadmap's own "shell + one real example, not every panel" scope for this handoff.
- **`BH_Commerce`** (`own-ur-shit/includes/class-commerce.php`) — wraps WooCommerce product upsert/read and order/subscription status behind `upsert_product()`, `get_product()`, `get_order()`, `is_subscription_active()`, etc. Migrated consumer: `bh-monetization-woo`'s `BHM_Products::sync_tier_wc_product()` and `sync_object_purchase_product()` (`class-products.php`) now call `BH_Commerce::upsert_product()` instead of instantiating `WC_Product_*` directly, with the original direct-WooCommerce code kept as an explicit fallback if `BH_Commerce` isn't present (old core version) — same class_exists()-guarded-degrade convention as everything else here.
- **`BH_Media`/`BH_Mail`** — correctly not touched this pass, per the roadmap's own sequencing (low-priority, "slot in whenever convenient").

## Debug Tools additions this pass: copy-to-clipboard + edge-case seed presets

Two small, cross-cutting Debug Tools improvements, prompted by someone pasting a real Test Runner failure list into a session and wanting a faster way to do that:

- **Copy-to-clipboard on both the Test Runner and the Console/Logs sections** (`OUS_TestRunner`/`OUS_DebugLog`, both in own-ur-shit). Test Runner gets a "Copy ALL failures" button (every failing row, across every suite) plus one "Copy failures (N)" button per suite — deliberately FAILURES ONLY, not a full dump, since that's the thing worth pasting into a chat/ticket. The Console/Logs page gets one "Copy N log entries" button for whatever's currently visible (respects the existing level filter). Both share one `window.bhCopyToClipboard(textareaId, btn)` JS helper (async Clipboard API with an `execCommand('copy')` fallback), guarded against being defined twice since both sections can render on the same Debug Tools page load.
- **Edge-case seed presets** (`BHC_Debug::edge_case_presets()` in bh-courses, the reference implementation of this pattern) — a dropdown of NAMED scenarios (empty lesson, unknown/missing step type, quiz boundary values, zero-question quiz, missing passing_score, invalid video URLs, invalid image attachment IDs, an XSS attempt in a text step) that seed exactly the malformed/boundary step data `BHC_Steps::save()` is supposed to defend against — the same shapes the PHPUnit/Test Runner suites already assert on, now clickable in the real admin UI instead of only asserted in code. This is the pattern every other plugin's own debug-seed section should grow toward next: not just "seed one realistic happy-path fixture," but a picker of named edge cases specific to whatever that plugin's own validation/sanitization logic actually has to get right (bh-contest: vote-limit boundary, expired-contest voting attempt; bh-registry: malformed submitted URLs, duplicate link detection; bh-monetization-woo: refund/chargeback timing races). Only bh-courses got the full treatment this pass — flagged here so it isn't lost as a "do this everywhere" follow-up rather than assumed already done.
- Also fixed while building the video-URL edge-case preset: `BHC_Steps::save()`'s video-URL validation only ever ran the string through `esc_url_raw()` (sanitizes characters, doesn't validate shape), so a value like `not a url` silently passed through as `http://not%20a%20url` instead of being dropped. Added a `filter_var($url, FILTER_VALIDATE_URL)` check alongside the existing `esc_url_raw()` call — a real bug this pass's own edge-case-preset work surfaced, not a hypothetical.
