# Design Suite unification & the two new top-level menus — pulling Style + the Element Builder + the CRM out of Debug Tools and into real, capability-gated employee-facing product

A design pass for the request AJ made directly, across several messages, verbatim:

> "I also feel like the element builder deserves to exist in its own menu outside of debug. It should be usable by real users, not just like devs and admins" / "Meaning like real employees not front end users" / "So like ones team can go in and build out their shit — its a full ass feature, not something to tuck away." / "Same with CRM, it should have like the people and profiles, and the project tracks and other eventual submenues." / "As such, style and the element builder and all that stuff should be in the same menu" / "But not all under the Own Ur Shit menu..." / "Also the UX really sucks on the element building... make it feel like the style editor exactly... it should feel like a nice little integrated inspector based builder and design suite for the whole site." / "Like the style interface IS the builder interface really, it just needs per 'thing' styles, and a way to like add nested elements easily, and build and style out whole interfaces in a very Godot like way... and databinding needs to be wayyyyy better and more flexible at runtime" / "(there should still be separation between what any given plugin adds to the style inspector and the element builder custom elements)"

This doc is grounded in the code that already ships, cited by file and class, in the same call-site-first spirit as `ELEMENT-BUILDER-DESIGN-PLAN.md` and `LMS-AUTHORING-DESIGN-PLAN.md`. It is the direct successor to `ELEMENT-BUILDER-DESIGN-PLAN.md`: that doc built the *engine* (`BH_Element`, `BH_Element_Data`, `BH_Element_Prefab`, the `bhcore_element_placements` table, the `BHY_Gallery`-cloned three-pane `BH_Element_Builder` GUI) and shipped it as a Debug Tools section. This doc is about **promoting that engine into a real, employee-facing product surface, merging it with the Style editor into one inspector, and materially upgrading data binding** — without reopening any of the engine-level judgment calls that doc already settled.

## 0. The findings that frame the design

Four facts from reading the actual code determine almost everything below:

1. **The "make it feel like the style editor" ask is nearly free, because the Element Builder was already built as a `BHY_Gallery` clone.** `ELEMENT-BUILDER-DESIGN-PLAN.md` §4 explicitly cloned `BHY_Gallery`'s three-pane layout (`class-style-gallery.php`: grouped sidebar / iframe canvas with a `#bhy-vars` live-rewrite `<style>` tag / `BHY_UI` picker controls) into `BH_Element_Builder`. The two interfaces are already siblings. The user's "the style interface IS the builder interface really" is not a rebuild request — it is a request to **finish collapsing two already-similar three-pane screens into one**, add the one thing the element side is missing (per-instance style + nested-tree authoring in the same inspector), and stop hiding it in Debug Tools.

2. **Per-instance style has a data slot already reserved but unused.** `BH_Element::render_placement()` (class-element.php lines 442-447) already builds an `$instance` array containing `'style' => is_array($config['style'] ?? null) ? $config['style'] : []` and hands it to every type's `render` callable — but **nothing currently reads or applies it.** `config.style` is a designed-in, documented, currently-dead field on every placement. The "per 'thing' styles" ask is: give that field a cascade and a wrapper, and expose it in the inspector with the *same* `BHY_UI` pickers the Style editor uses. The storage seam already exists; no schema change is required for v1.

3. **The standalone-page bug is real, unresolved, and specifically diagnosed — and the working pattern is known.** `own-ur-shit.php`'s version log (3.4.8-3.4.13) and `class-api-docs.php`'s docblock record a multi-pass, never-root-caused incident where `admin.php?page=ous-api-docs` and `?page=ous-codebase-docs` were denied access ("Sorry, you are not allowed to access this page") *despite* `add_submenu_page()` returning a real hook_suffix and `current_user_can('manage_options')` reading TRUE on the failing request. The decisive finding (3.4.11): **the only two pages in the entire ecosystem that failed were the only two that wrapped their `add_submenu_page()` call in a conditional (`is_locked()`) before registering.** Every unconditionally-registered page — the `own-ur-shit` top-level menu, the `ous-debug` top-level menu, Style, Security, Content Studio, and every peer plugin's own screens — works. This gives us a concrete, mirror-the-working-pattern mitigation (§1.4), not a vague "watch out."

4. **The role/capability infrastructure for "real employees, not just admins" already exists and runs early enough.** `OUS_Roles` (class-roles.php) registers granular capabilities to roles via the `bhcore_role_capabilities` filter on `init` — *before* `admin_menu` fires. It already ships `bhcore_manage_students` and `bhcore_review_submissions` and explicitly documents that "any plugin can register its OWN new capability the same way, via the same filter, with zero changes needed here." The employee-access story is a new capability registered through this exact mechanism, not a parallel permission system (§1.3).

Net: this is a **reorganization + unification + a targeted binding upgrade**, not a new engine. Every existing contract (`BH_Element`, `BH_Element_Data`, `BH_Element_Prefab`, `BHY_Style`, `BHY_Gallery`, `BHY_UI`, `BH_Content`/`BH_Studio`, the `ous/v1/elements/*` REST bridge, `OUS_MenuMerge`, `OUS_Roles`) stays intact.

## 1. Menu restructure

### 1.1 The two new top-level menus

**"Design Suite"** (slug `bh-design`) and **"CRM"** (slug `bh-crm-hub`) become two new top-level `add_menu_page()` entries, siblings of the existing "Own Ur Shit" (`own-ur-shit`, position 3) and "OUS Debug" (`ous-debug`, position 99) top-level menus. Naming: "Design Suite" is recommended over alternatives ("Studio", "Builder", "Design") because (a) it reads as employee-facing product, not dev tooling; (b) "Studio" already means `BH_Studio` (the Gutenberg-primitives canvas) internally and would collide; (c) it accurately covers both site-wide style tokens AND per-element composition. If AJ prefers a shorter label, "Design" is the fallback — the *slug* (`bh-design`) should not change regardless, since slugs are load-bearing for the REST/hash-link plumbing below.

The existing "Own Ur Shit" menu stays exactly what it is: the **ecosystem activation/installation dashboard** (`OUS_Dashboard`, the registry cards + system status). The user's "but not all under the Own Ur Shit menu" is honored literally — Design and CRM leave that menu; it keeps only what is genuinely "install/activate the ecosystem" shaped. `OUS_Debug` stays a separate top-level dev menu, unchanged.

#### Design Suite (`bh-design`) submenus

| submenu slug | label | callback (existing class, relocated) | notes |
|---|---|---|---|
| `bh-design` | Design Suite | `BH_Design_Suite::render()` (**new thin shell**, §2) | the unified inspector — Style tokens + element authoring in one three-pane screen |
| `bh-style` | Site Styles | `BHY_Gallery::render()` (**existing**) | kept as its own submenu during transition (§2.4); the global-token authoring surface |
| `bh-studio` | Content Studio | `BH_Studio::render()` (**existing**) | the Gutenberg-primitives block canvas, moved here from under `own-ur-shit` — it is a "design the site" tool, not an activation tool |
| `bh-prefabs` | Prefabs | `BH_Element_Prefab::render_manager()` (**new thin listing**, optional) | a real management view for the already-shipped prefab system, currently only reachable inside the builder GUI |

#### CRM (`bh-crm-hub`) submenus

| submenu slug | label | callback | notes |
|---|---|---|---|
| `bh-crm` | People | `BHCRM_People::render()` (**existing**) | the person list / profile detail; keeps its existing slug so every `admin.php?page=bh-crm&...` deep link (profile links, `dashboard_link`) keeps working unchanged |
| `bh-crm-projects` | Project Tracker | `BHCRM_People::render()` w/ a forced `project_id`/board dispatch, OR a new `BHCRM_Projects::render_boards()` list (§1.5) | the kanban boards; today reached only via a dispatch branch inside the People page |

The `bh-crm-hub` parent slug is deliberately **not** `bh-crm` (that stays the People page's own slug). WordPress requires the top-level menu's own slug to equal its first submenu's slug to avoid a duplicated first item; we solve that the same way `OUS_Dashboard::add_menu()` already does (§1.4) — register the top-level with a distinct hub slug whose callback redirects/renders People, then add People as the first real submenu. Simplest robust option: make the `bh-crm-hub` top-level callback identical to `BHCRM_People::render()` and set the first submenu's slug to `bh-crm-hub` too, exactly mirroring the `own-ur-shit` pattern (a top-level + a same-slug submenu relabeled). People's *deep-link* slug `bh-crm` is then added as the second submenu. Both slugs render `BHCRM_People::render()`; this is the same "register the real callback twice under different slugs, never rely on empty-callback slug reuse" rule `OUS_MenuMerge`'s docblock already establishes.

### 1.2 Interaction with `OUS_MenuMerge`

`OUS_MenuMerge` (class-menu-merge.php) currently relocates peer-plugin custom pages *under* `own-ur-shit` at `admin_menu` priority 999, driven by the `admin_menus` entries in `OUS_Registry` (class-registry.php: bh-crm declares `['slug'=>'bh-crm','label'=>'People/CRM','callback'=>['BHCRM_People','render']]`; bh-registry and bh-monetization-woo declare their own).

**Decision:** extend `OUS_Registry`'s `admin_menus` entry shape with an optional `'parent'` key (default `'own-ur-shit'`), and teach `OUS_MenuMerge::merge()` to honor it. bh-crm's registry entry changes its `parent` to `bh-crm-hub`; a new Design-oriented entry (or direct registration) targets `bh-design`. This keeps the ecosystem's "one place decides where a plugin's pages live, plugins stay unaware" property — the user's restructure is expressed as data in the registry, not scattered `add_submenu_page` edits across plugins. `OUS_MenuMerge` must register the two new **parent** top-level menus itself (or a small new `BH_Design_Suite::add_menu()` / `BHCRM_Hub::add_menu()` does) *before* priority 999 so the relocation targets exist.

Sequencing hazard to respect: the parent `add_menu_page()` calls must run at the **default** `admin_menu` priority (10), and the merge/relocation at 999, so the parents exist when submenus attach. This matches how `own-ur-shit` (default priority, `OUS_Dashboard::add_menu` hooked normally) and the 999 merge already coexist.

### 1.3 Capability story — "real employees, not just devs/admins"

Every existing admin page in this ecosystem gates on `manage_options` (admin-only): `BHY_Gallery`, `BH_Studio`, `BHCRM_People`, all of `BH_Element`'s REST routes. That is exactly the "just devs and admins" the user is pushing against.

**Decision:** register **two new capabilities** via the existing `OUS_Roles` / `bhcore_role_capabilities` mechanism (class-roles.php), granted to `administrator` by default (so nothing an admin can do today changes):

| capability | gates | rationale |
|---|---|---|
| `bhcore_design_site` | the Design Suite menu + all `ous/v1/elements/*` and `bhy` style-save routes when invoked from the suite | "an employee who builds/styles interfaces" — the element-builder-and-style audience the user named ("ones team can go in and build out their shit") |
| `bhcore_manage_crm` | the CRM menu (People + Project Tracker) | "an employee who works the CRM/project boards" — distinct from designing the site; a project manager is not necessarily a site designer |

Both are added to `OUS_Roles::DEFAULT_CAPS` with `['administrator']`, and — because the user explicitly wants *employees* — the recommended default also grants them to `editor` (a real, pre-existing WordPress role that already means "trusted non-admin staff"). Concretely: `'bhcore_design_site' => ['administrator','editor']`, `'bhcore_manage_crm' => ['administrator','editor']`. A site that wants a narrower "designer but not editor" account grants the bare cap to any role via the existing `bhcore_role_capabilities` filter or a one-off `$user->add_cap()`, with no new UI required — exactly the unlock `OUS_Roles`'s own docblock already describes for `bhcore_manage_students`.

**Why not `edit_posts`:** `edit_posts` conflates "can write blog posts" with "can design the site / work the CRM," and can't be independently revoked. A purpose-named capability is the ecosystem's own established idiom (`bhcore_manage_students`) and is what lets Billy hand a contractor CRM access without also handing them the site's design system or post authoring.

**Capability enforcement must be consistent across menu AND REST.** Today the REST routes hardcode `current_user_can('manage_options')`. The Design Suite's write path (saving placements, saving per-instance style) must accept `current_user_can('bhcore_design_site') || current_user_can('manage_options')`. This is a real, security-relevant change: loosening the `ous/v1/elements/*` permission callbacks is what actually makes employee authoring work, and it must be done deliberately, not by accident. See §5 out-of-scope for the front-end (non-admin, non-employee) boundary that stays closed.

### 1.4 Standalone-page bug mitigation — mirror the working pattern exactly

The incident (finding 0.3) gives a concrete, testable mitigation. The new top-level menus must mirror the **unconditionally-registered, default-priority, real-callback** pattern that every working page in this ecosystem uses, and specifically must NOT reproduce the one thing the two broken pages did.

Rules the new registration must follow (each traced to the incident log):

1. **Register unconditionally.** Never wrap `add_menu_page()`/`add_submenu_page()` in `is_locked()`, environment checks, or any conditional (3.4.11's decisive finding: conditional registration was the single differentiator of the failing pages). Gate *behavior* (what the page shows / a destructive action) inside the render callback, never the *registration*.
2. **Register at default `admin_menu` priority (10)**, like `OUS_Dashboard` and `OUS_Debug` — not an early or unusual priority. (3.4.8 showed an early-priority side effect — a synchronous rewrite flush at `init` priority 10 — cascaded into API Docs 404s; keep registration boring and on-time.)
3. **Pass the real callback**, never an empty string relying on slug reuse (`OUS_MenuMerge`'s docblock: empty-callback slug reuse "does not reliably work for custom pages" because the internal hook name derives from slug + parent).
4. **Gate on a capability the target user actually has at `admin_menu` time.** `add_submenu_page()` marks a page `$_wp_submenu_nopriv` if `current_user_can()` fails *at registration moment* (3.4.13's root-cause identification of the `$_wp_submenu_nopriv` mechanism). Because `OUS_Roles` grants the new caps on `init` (before `admin_menu`), an employee's `bhcore_design_site`/`bhcore_manage_crm` cap is already present when the menu registers — this is why the capability must be a real granted cap, not a runtime-computed check.
5. **Prefer a top-level menu over a submenu-of-a-submenu.** Both pages that failed were *submenus of `ous-debug`* (a submenu chain). Both top-level menus that exist (`own-ur-shit`, `ous-debug`) work flawlessly. The new Design Suite and CRM being **top-level** menus (which the user wants anyway) puts them squarely in the proven-working category, not the failure-prone one.
6. **Log the registration result** (hook_suffix vs FALSE) via `OUS_DebugLog::log_throttled()` on first registration, exactly as `class-api-docs.php` and `class-codebase-docs.php` now do, so if the bug recurs on these menus it is immediately diagnosable rather than a silent 404.

**Deployment risk to watch (flag, do not hand-wave):** this WordPress-core-level access-denial has never been root-caused across five-plus diagnostic passes on this specific install, and appeared correlated at least once with OPcache serving stale compiled menu code (3.4.10) and once with a mid-request `wp_cache_flush()` (3.4.8). The first live smoke test of the new menus MUST be: log in as a *non-admin employee account* holding only `bhcore_design_site`/`bhcore_manage_crm` (not as an admin, whose `manage_options` masks capability-scoping bugs) and confirm both top-level menus render their real content, on a request after an OPcache reset. If either 404s despite the six rules above, the fallback is the same one the ecosystem already chose for API/Codebase Docs: surface the content as a section on an already-proven-loading page. That fallback should be considered a real possibility for these menus, not an unthinkable one — hence keeping the Debug Tools element-builder section alive during transition (§4).

### 1.5 The Project Tracker submenu

Today the kanban board is reached only through a `project_id` dispatch branch inside `BHCRM_People::render()`/`render_detail()` (bh-crm.php 1.2.0 changelog; class-projects.php). To make it a first-class CRM submenu that "more submenus can be added later without restructuring," add a thin `BHCRM_Projects::render_boards()` that lists all `bhcrm_projects` rows and links each to its board (the existing dispatch), registered as the `bh-crm-projects` submenu. The People page's existing "Projects" section and its `project_id` dispatch stay as-is (a board is still reachable from a person too). No change to the board data model, the `bhcrm_project_board` surface, `bh/sticky-card`, or the `ous/v1/elements/placements/*` save path. Future submenus (Notes, Reports, Tags) slot in as additional `admin_menus` registry entries with `'parent' => 'bh-crm-hub'` — zero restructuring, which is the extensibility the user asked for.

## 2. The unified Style + Element inspector

### 2.1 What "the style interface IS the builder interface" means concretely

`BHY_Gallery` (Style) and `BH_Element_Builder` (elements) are already both three-pane `left rail / iframe canvas / right controls` screens built from the same `BHY_UI` pickers and the same `#bhy-vars` live-rewrite technique (`class-style-gallery.php` render_script; `ELEMENT-BUILDER-DESIGN-PLAN.md` §4 status note 2026-07-11). The unification is a **new thin shell class `BH_Design_Suite`** (`own-ur-shit/includes/class-design-suite.php` + `assets/js/design-suite.js` + reused CSS) that presents ONE screen with:

- **Left rail — a mode-aware navigator with two stacked regions:**
  - a **surface switcher** (top): `Site (global tokens)`, `Dashboard`, `CRM profile`, `Portal panel`, `Lesson UI`, each CRM project board, etc. — populated from `GET ous/v1/elements/surfaces` plus a synthetic "Site" entry for global-token-only mode. Choosing "Site" collapses the tree and shows only global token controls (i.e. the current `BHY_Gallery` behavior, embedded).
  - a **Godot-style scene tree** (below the switcher, §2.4): the placement/child hierarchy of the currently selected surface+context, rendered as an indented, selectable, reorderable outline. This replaces `BH_Element_Builder`'s flat palette-first flow with a tree-first one.
- **Center — the live iframe canvas**, unchanged in technique from both existing screens: `<iframe srcdoc>` with the surface's CSS + a `#bhy-vars` `<style>` tag (global tokens) + the surface's slots rendered via `POST ous/v1/elements/preview`. Editing a *global* token in the right rail live-rewrites `#bhy-vars` across the frame exactly as `BHY_Gallery::render_script()`'s `refreshAllFrames()` already does. Editing a *per-instance* override live-rewrites that element's own scoped `<style>` block (§2.3).
- **Right rail — the unified inspector**, whose contents are **contextual to the left-rail selection**:
  - **Nothing selected / "Site" mode** → the global token controls: the exact `BHY_UI::swatch_field`/`font_field`/`slider_row` set `BHY_Gallery::render_controls()` renders today, saving through the existing `admin-post bhy_save_settings` path. This is the Style editor, in-place.
  - **A placement/element node selected** → three stacked inspector sections, matching `ELEMENT-BUILDER-DESIGN-PLAN.md` §4's inspector but now with a real Style section:
    1. **Element** — its type, and one control per `schema` attr (text/number/checkbox by `type`; richtext/container attrs get an "Edit content →" button opening the existing `BH_Studio` canvas at `('bh_element', content_context_id)`).
    2. **Style (per-instance overrides)** — for each token in the element type's `'style'` manifest list (e.g. `bh/stat-card` declares `['color_accent','radius','space_scale']`), the SAME `BHY_UI` picker the global editor uses, but writing into this placement's `config.style` map (§2.3) instead of the global option.
    3. **Data** — for each `bindable` attr, the binding editor (unchanged from today in v1; upgraded in §3): source dropdown from `GET ous/v1/elements/sources?kind=`, arg fields, context-token selector, Bind/Literal toggle, inline "resolves to:" preview via `POST preview`.

### 2.2 The plugin-separation requirement — two distinct registration surfaces, one inspector

This is the user's parenthetical — "there should still be separation between what any given plugin adds to the style inspector and the element builder custom elements" — and it is a hard architectural boundary this doc must protect against a future implementer collapsing.

A plugin contributes to the unified inspector through **two entirely separate registration APIs that must never merge**:

| contribution | registration surface | what it adds | example |
|---|---|---|---|
| **Style tokens / a previewable surface** | `add_filter('bhy_style_surfaces', ...)` (`BHY_Gallery`'s existing mechanism, class-style-gallery.php) | a rendered preview surface + (implicitly) participation in the global `--bh-*` token set | `BHC_StyleSurface::register()` registers `bh-courses-catalog`/`bh-courses-lesson` |
| **A placeable custom element type** | `BH_Element::register_type($slug, $args)` (class-element.php) | a new element in the palette, with its own `schema`/`style`/`surfaces`/`render` | bh-crm registers `bh/sticky-card`; bh-monetization registers storefront blocks |

These feed **one unified inspector UI** but remain **two independent inputs**:

- The global-token controls (right rail, "Site" mode) are driven by `BHY_Style::DEFAULTS` + the `bhy_style_surfaces` previews. A plugin adding a surface there does NOT thereby add an element.
- The palette / scene-tree element types are driven by `BH_Element::types_for_surface()`. A plugin adding an element type does NOT thereby add a global style surface.
- The per-instance Style inspector section (§2.3) reads an element type's `'style'` manifest — which is a **list of global token *names* the element opts into exposing per-instance**, NOT a new token registry. It reuses the same `--bh-*` vocabulary; it never mints element-scoped tokens. This is the seam that keeps the two systems related-but-separate: an element's per-instance style is always a *scoped override of the site's existing tokens*, so a plugin's element and a plugin's style surface share one token vocabulary without sharing a registration API.

**Explicit anti-goal for the implementer:** do NOT build a single `register_design_thing()` that takes both a style surface and an element type. Do NOT let `BH_Element::register_type()` auto-register a `bhy_style_surfaces` entry, or vice versa. A plugin that wants both calls both, guarded independently by `class_exists('BHY_Style')` and `class_exists('BH_Element')` respectively — so a plugin can contribute style tokens even where `BH_Element` is absent, and contribute elements even where the Style gallery is absent. Collapsing them breaks each plugin's ability to contribute to one system without the other.

### 2.3 Per-instance style overrides — data model & cascade

**Storage (v1, no schema change):** reuse the already-present-but-unused `config.style` map on each `bhcore_element_placements` row. `BH_Element::render_placement()` already decodes `config` and already passes `$instance['style'] = config['style']` to the render callable (class-element.php lines 396, 446). The map shape:

```json
{ "attrs": { ... }, "style": { "color_accent": "#ff3366", "radius": 4 } }
```

Keys are `BHY_Style` field names (the same keys `OVERRIDABLE_FIELDS` / the global option use — `color_accent`, `radius`, `space_scale`, etc.), constrained at save time to the element type's declared `'style'` manifest list (an element only stores overrides for tokens it opted into exposing). Values are validated with the existing `BHY_Style::safe_color()` / `BHY_Style::safe_number()` helpers — the same sanitizers the global save uses — so a per-instance override can never inject anything the global editor couldn't.

**Cascade / specificity (explicit, most-specific wins):**

```
per-instance override (config.style, this placement)
  >  element-type default   (the type's own baseline, if any — see note)
  >  entity/surface override (BHY_Style::entity_overrides(), if this surface has a post entity)
  >  site-wide global tokens (BHY_Style::get() / inline_css())
```

Implemented the same way `BHY_Style` already layers everything else: as **scoped CSS custom properties**. `BH_Element::render_slot()` wraps each placement in `<div class="bh-element" data-placement-id="N">` today (class-element.php line 485). Per-instance overrides render as a scoped block on that wrapper:

```html
<div class="bh-element" data-placement-id="42" style="--bh-accent:#ff3366;--bh-radius:4px;">…</div>
```

Because the element type's own stylesheet already reads `var(--bh-accent)`/`var(--bh-radius)` (the whole ecosystem's convention, class-style.php `inline_css()`), scoping those vars on the wrapper makes the override apply to that instance and its descendants only, inheriting the global `:root` values for every token it does NOT override. This is **the identical mechanism** `BHY_Style::entity_style_payload()`/`inline_css()` already use, applied at the element-wrapper scope instead of `:root` or a post entity. No new CSS system.

**New helper (small, justified):** `BHY_Style::scoped_inline_style(array $overrides): string` — takes a `{field => value}` map, returns a `style="--bh-x:…;"` attribute value string, running each value through the existing `safe_color`/`safe_number` by field name. `BH_Element::render_placement()` calls it on `$instance['style']` and applies the result to the wrapper. This is the one genuinely new piece of style code; it is a formatter over existing sanitizers, not a new cascade engine.

**Note on "element-type default":** an element type MAY declare baseline token values in its manifest (a future optional `'style_defaults'` key). Not required for v1 — omit it and the cascade collapses cleanly to instance-override > global. Named as a seam, not built now.

**Live preview:** editing a per-instance token in the right rail rewrites that one wrapper's inline `style` attribute in the canvas iframe (a tighter, per-element analogue of `refreshAllFrames()`), so per-instance restyling feels exactly like the global token sliders already feel.

### 2.4 The Godot-style nested-tree authoring UX

The user wants "a way to like add nested elements easily, and build and style out whole interfaces in a very Godot like way." Godot's editor is a **scene tree on the left, a viewport in the center, an inspector on the right** — which is precisely the three-pane shape already in play; the missing piece is the *tree* (today the left rail is a flat palette + a flat per-slot list).

**Left-rail scene tree — proposed structure and interactions:**

- **Roots = the surface's registered slots** (from `bh_element_surfaces`, e.g. CRM profile's `header`/`main`/`sidebar`). Each slot is a non-deletable tree root.
- **Children of a slot = its placements** (from `get_placements(surface, context, slot)`, ordered by `position`).
- **Children of a container placement** (`bh/container`, `bh/sticky-card`, any `container:true` type) = the nodes of its embedded `BH_Content` tree at `('bh_element', content_context_id)`, read via `BH_Content::get()`. This reuses the EXACT hybrid-nesting bridge `ELEMENT-BUILDER-DESIGN-PLAN.md` §1.1 designed and `bh/container` already proved — the tree simply *visualizes* the two-level nesting (placement-level slots/elements, then `BH_Content` children inside containers) that already exists, rather than inventing new nesting.

**Interaction flows (spec-level, so the build pass doesn't re-decide):**

- **Select a node** → right-rail inspector switches to that node's Element/Style/Data sections (§2.1). Selecting a slot root shows a slot-level panel (add-child button + slot label only). Selecting a `BH_Content` child node routes editing into the embedded `BH_Studio` canvas for that container (the node is a link into Studio, not editable inline — no second block editor is built, per `LMS-AUTHORING-DESIGN-PLAN.md`'s standing rule).
- **Add a child** → a `+` on any slot root or container node opens an inline palette filtered by `types_for_surface(surface)` (for slot-level adds) or the container's allowed `BH_Content` block types (for inside-container adds). Choosing a type inserts a placement (via the existing `POST ous/v1/elements/placements/{surface}/{context}` full-slot upsert) or a `BH_Content` child (via the existing `ous/v1/studio/bh_element/{id}` save) — no new endpoints. This satisfies "add nested elements easily... in that same panel (not a separate 'add element' flow)": the add control lives *on the tree node*, inline.
- **Reorder** → drag a node within its parent; on drop, re-POST the affected slot's placements in new array order (the existing upsert route sets `position` from array index — `rest_save_placements()`), or re-save the container's `BH_Content` children order. Same mechanism `kanban-board.js` already uses for card moves.
- **Style-override editing** → happens in the right-rail Style section for the selected node (§2.3); live-updates the node's wrapper in the canvas.
- **Binding editing** → right-rail Data section for the selected node (§3).
- **Delete** → the real `DELETE ous/v1/elements/placements/{id}` route (already shipped, class-element.php lines 556-571) for placements; `BH_Content` child removal via Studio save for inner nodes.

**Reuse, not rewrite:** every action above maps to an already-shipped REST route. The scene tree is a new *presentation and selection layer* over the existing placement + `BH_Content` bridges, exactly as the kanban board is a bespoke presentation over the same bridge. `design-suite.js` owns the tree; it invents no storage.

### 2.5 Responsive UX — a quality experience on both mobile and desktop

The three-pane shell (§2.1) is a desktop-first shape; it must degrade to a genuinely usable one-column experience on narrow viewports rather than shrinking three rails into unusable slivers. The discipline here is **follow WordPress admin's own responsive conventions, don't reinvent them** — WP admin already collapses its sidebar to the hamburger toggle at ≤782px (the `.auto-fold`/`.folded` behavior) and already stacks meta boxes into a single column at that same breakpoint. The Design Suite should key off the same breakpoint and feel native to it.

**Breakpoints and layout behavior:**

- **≥1200px (desktop, default):** the full three-pane layout — left scene-tree/surface rail (fixed, ~260–300px), center live iframe canvas (fluid), right inspector rail (fixed, ~320–360px). Unchanged from §2.1.
- **783–1199px (tablet / small desktop):** keep three panes but narrow the side rails; below ~1000px, collapse the **left rail to an icon/label toggle** (a "Tree" button that slides the scene tree over the canvas as a temporary overlay, dismissed on selection), mirroring WP admin's own `.folded` sidebar collapse.
- **≤782px (mobile, matches WP admin's own breakpoint):** panes **stack to a single column**. The surface switcher + scene tree become a **collapsible top section** (compact dropdown + disclosure-toggled tree, collapsed by default so the canvas is visible on load). The canvas is the primary, full-width column. The **inspector becomes a bottom sheet / slide-over** — selecting a node slides the Element/Style/Data inspector up from the bottom over the canvas, with a drag handle and dismiss control, instead of a permanent right rail. Standard mobile inspector idiom; reuses the same `BHY_UI` picker markup, only the container chrome changes.

Implement with plain CSS media queries plus a small amount of `design-suite.js` state for the overlay/bottom-sheet open/close — no separate mobile codebase, consistent with the ecosystem's no-build-step convention.

**Touch targets and reordering.** The existing `BH_Element_Builder` inspector already reorders placements with **up/down buttons, not drag-and-drop** — already mobile-friendly. The new Godot scene tree (§2.4) proposes drag-to-reorder as its primary gesture, which is fiddly on touch (long-press, imprecise drop targets, scroll-vs-drag ambiguity). **The scene tree must not ship drag-only** — every reorderable node also exposes up/down (move-within-parent) controls, inline on desktop hover and always-visible on touch, calling the identical re-POST-in-new-order route as drag. All interactive controls should meet a minimum ~44px touch target on mobile.

This responsive discipline extends to the CRM **kanban board** and the **portal panel** built in prior passes (not redesigned here, but held to the same standard): kanban columns must remain reachable via horizontal scroll or a stacked-lane fallback on narrow screens, and portal-panel content reflows to one column at the same ≤782px breakpoint.

### 2.6 A comprehensive, Tailwind-inspired preset control library

§2.3 gave per-instance style a storage seam (`config.style`) scoped to a handful of `--bh-*` tokens. To be genuinely comprehensive, the inspector's Style section should expose **preset-scale controls for essentially every layout-relevant CSS property**, staying faithful to the ecosystem's existing constrained-scale philosophy rather than inventing a second scale.

**Design principle — presets by default, freedom on demand.** Every property control offers a **preset-scale picker** (fast, consistent, Tailwind-utility-inspired — exactly the ecosystem's existing instinct with `space_scale`/`font_scale`/`radius`) AND a **per-property escape hatch to a raw custom value**. Presets keep most edits on-scale and consistent; the escape hatch means no value is ever unreachable.

**Grounding in what already exists.** `space_scale` (0.6–1.8), `font_scale` (0.75–1.6), `radius`/`radius_sm` (0–32px), and the `--bh-*` color token vocabulary (`class-style.php` `DEFAULTS`/`OVERRIDABLE_FIELDS`) are the existing constrained scale. The preset library reuses these: colors always pick from existing `BHY_Style` color tokens (per §2.2's plugin-separation principle — never a new token registry), spacing/radius steps are anchored to the existing `--bh-space-scale`/`--bh-radius` system.

**Property groups the Style section should cover** (`[core]` = earlier phase, `[adv]` = later/lower-priority phase):

| group | properties | preset scale (example steps) |
|---|---|---|
| **Sizing** `[core]` | width, height, min/max-width/height | `auto / 0 / xs / sm / md / lg / xl / full / screen` + custom |
| **Spacing** `[core]` | margin, padding (all sides + per-side) | `0 / xs / sm / md / lg / xl` anchored to `--bh-space-scale` + custom |
| **Background** `[core]` | color (existing tokens), image, size, position, repeat | color = `@token:*`; size = `auto/cover/contain` + custom |
| **Typography** `[core]` | font-size, weight, line-height, letter-spacing; color via tokens | size anchored to `--bh-font-scale`; weight `400/500/600/700` + custom |
| **Border** `[core]` | width, style, color (tokens), radius | width `0/1/2/4`; radius anchored to `--bh-radius` scale + custom |
| **Display / layout** `[adv]` | display type; flex (direction, wrap, justify, align, gap); grid (cols, gap) | display `block/flex/grid/inline-block/none`; gap on space scale + custom |
| **Position** `[adv]` | position type, offsets, z-index | `static/relative/absolute/sticky/fixed`; z `0/10/20/30/40/50` + custom |
| **Effects** `[adv]` | opacity, box-shadow presets, simple transform presets | opacity `0–100`; shadow `none/sm/md/lg` + custom |
| **Overflow / visibility** `[adv]` | overflow(-x/-y), visibility | `visible/hidden/scroll/auto` + custom |

**HTML attributes — simpler, and per-element-type, not universal.** A parallel, lighter inspector section exposes common attributes: `id`, additional `class` tokens, `title`, `aria-label`, custom `data-*` attributes, and element-appropriate ones like `href`/`target`/`rel` (link-capable elements) and `alt` (images). Which attributes are relevant is declared **per element type at registration**, extending `BH_Element::register_type()`'s `$args` contract with an optional `'attrs'` key:

```php
'attrs' => [
    'id'         => true,
    'class'      => true,
    'aria-label' => true,
    'href'       => ['type' => 'url'],     // only link-capable types declare this
    'target'     => ['enum' => ['_self','_blank']],
    'data-*'     => true,                   // opt into the custom data-attr UI
],
```

An element with no `'attrs'` key gets a minimal default (`id`, `class`, `aria-label`); it never gets `href`/`alt` unless it opts in — mirroring the existing `'style'` manifest pattern exactly.

**Custom `data-*` attributes, made genuinely usable, not just a raw key/value pair.** Beyond the blanket `'data-*' => true` opt-in above, the inspector's custom-attribute UI should support a **repeatable key/value row list** (add/remove rows, each row = one `data-<key>` = `<value>` pair), with light per-row typing where useful (a "value type" hint — text / number / boolean-as-`"true"/"false"` / JSON — purely for input-affordance and validation, since HTML attribute values are always ultimately strings). An element type can optionally pre-declare **named, structured** data-attributes with real semantics (e.g. `bh/sticky-card` pre-declaring `data-status` as an enum picker rather than free text) by listing them explicitly in `'attrs'` (`'data-status' => ['enum' => [...]]`) — the blanket `'data-*' => true` remains the fallback for anything not pre-declared, so authors are never blocked from adding an arbitrary custom attribute even on a type that didn't anticipate it. Stored in the same sibling `htmlAttrs` map (below), namespaced under a `custom` sub-key to keep pre-declared and freeform data attributes visually and structurally distinct in the inspector: `htmlAttrs.custom = {"data-role": "kpi", "data-status": "in_progress"}`.

**Semantic HTML tag selection — elements are not all `<div>`.** Every element type's render wrapper today is implicitly a generic container. The inspector should let an author choose the **semantic HTML tag** the element renders as, where more than one tag is sensible for that type — most usefully: interactive/action elements as `<button>` or `<a>` (with `href`/`target` attrs, from the section above, only relevant once `<a>` is chosen), and structural/content elements as `<section>`, `<article>`, `<aside>`, `<nav>`, `<header>`, `<footer>`, `<ul>`/`<li>` (for list-shaped content), or `<figure>`, rather than always `<div>`/`<span>`. Concretely:

- Extend `register_type()`'s `$args` with an optional `'tags'` key: an allowed-tag list for that type, with the first entry as the default, e.g. a generic container type might declare `'tags' => ['div','section','article','aside']`, an action/CTA-shaped type might declare `'tags' => ['button','a']`. A type that doesn't declare `'tags'` defaults to `['div']` (today's implicit behavior, unchanged — this is additive, not a breaking change to existing element types).
- The chosen tag is stored as a single `htmlAttrs.tag` value (falls back to the type's default when absent — old placements created before this feature ships need no migration).
- `render_placement()` reads `htmlAttrs.tag` (validated against that type's declared `'tags'` allowlist server-side — never trust an arbitrary client-supplied tag name) and emits that tag for the wrapper instead of a hardcoded `div`, carrying the same `data-placement-id`/style/attrs it already emits.
- Choosing `button` or `a` should sensibly cross-wire with the attribute section above: selecting `a` reveals the `href`/`target`/`rel` attribute controls (already declared per-type in `'attrs'`) in context, rather than showing them permanently regardless of tag choice. This keeps the inspector from showing irrelevant controls.
- This is deliberately scoped to *tag choice*, not a general HTML-templating system — the semantic tag changes the wrapper element only; the element's actual rendered content/children are unaffected by this choice.

**How this maps into `config.style`/`htmlAttrs` storage — concrete convention.** `config.style` gains **namespaced `group.property` keys** alongside the existing bare token-name keys from §2.3:

```json
{
  "style": {
    "spacing.padding":   "md",
    "sizing.width":      "full",
    "bg.color":          "@token:color_accent",
    "border.radius":     "sm",
    "type.font-weight":  "600",
    "sizing.height":     "custom:320px"
  },
  "htmlAttrs": {
    "tag": "a",
    "id": "hero", "aria-label": "Primary stat",
    "href": "/pricing", "target": "_self",
    "custom": { "data-role": "kpi", "data-status": "in_progress" }
  }
}
```

A `style` value is either a preset-scale step (`"md"`, `"full"`), a `@token:<name>` reference to an existing color token (colors are always token refs, never raw hex), or a `custom:<raw CSS value>` escape hatch. `htmlAttrs` is a sibling map (not nested under `style`) since HTML-attribute sanitization policy differs from CSS-value sanitization.

**Rendering.** Extend §2.3's `BHY_Style::scoped_inline_style()`: bare-token keys still emit `--bh-*` custom properties (unchanged); `group.property` keys resolve their preset step to a concrete CSS declaration (group scale table lookup, `@token:*` → `var(--bh-*)`, `custom:*` → sanitized literal) emitted as direct declarations on the same wrapper. Each value is sanitized by group-appropriate validators (extending `safe_color`/`safe_number` with `safe_length`/`safe_enum`). `htmlAttrs` (including `tag`) are applied by `render_placement()`/`render_slot()` through a strict per-attribute, per-type allowlist — never emitting an attribute or tag the type didn't declare, `data-*` values escaped, `href`/`target`/`rel` policy-checked. Still one wrapper, still scoped, no new CSS system.

**Phasing — superseded 2026-07-11 per AJ's explicit instruction ("I dont want suggestions deferred, Id like all the controls this go around"):** ALL property groups (`[core]` and `[adv]` alike — sizing, spacing, background, typography, border, display/flex/grid, position, effects/transforms, overflow/visibility), the full custom `data-*` row-list editor, per-type structured data-attribute declarations, and semantic-tag selection all ship together in Phase 2 (§4). The `[core]`/`[adv]` tags above are kept only as a reading aid for which groups are simpler vs. more involved — they no longer indicate a build-order split, and Phase 4a (originally "advanced groups, deferred") is removed from §4 accordingly.

## 3. Runtime data binding — "wayyyyy better and more flexible at runtime"

### 3.1 Today's contract and its four concrete limits

`BH_Element_Data` (class-element-data.php) today: an attr value is `{"literal": X}` or `{"bind": {"source": slug, "args": {...}, "subject": "context.user_id"}}`. `resolve()` looks up one registered source, substitutes `context.*` tokens from a server render context, calls the source's `resolve` callable, and falls back to the literal/default on any failure. It is robust and well-degraded — but it is:

- **(a) Server-side-only, render-time-only.** Everything resolves once, in PHP, when the surface renders. A dashboard stat-card showing "plays in last 30 days" is frozen at page load; nothing can re-fetch without a full reload. "At runtime" / "more flexible at runtime" most naturally means: the frontend should be able to re-resolve a binding after load.
- **(b) Single, flat, un-composable.** One binding = one source + static args. No formatter/transform on the output (a raw count can't be rendered as "1.2k" or a date can't be humanized), and one binding's args can't reference another binding's resolved value.
- **(c) Context-limited to server-scoped tokens.** `context.*` only reaches `user_id`/`post_id`/`entity_id`/`viewer_id` — server-known values. It cannot bind to sibling/parent element state, or to live client-side state (a form input, a selected tab, an interaction).
- **(d) No client mirror.** There is no JS equivalent of `resolve()`, so even the data the server already has can't be re-applied client-side without a round trip that re-renders whole HTML.

### 3.2 The upgrade — phaseable, v1 much better than today, clear path beyond

**v1 (ship first, highest value/lowest risk): client-side re-resolution over the existing REST bridge + output formatters.**

- **Runtime re-resolution.** Add `GET/POST ous/v1/elements/resolve` — body `{ "bindings": [ {"source","args","subject"}... ], "ctx": {...} }`, returns `[{value}...]`. This is a thin HTTP wrapper over the *existing* `BH_Element_Data::resolve()` (server stays the single source of truth for what a source returns and how it's sanitized — no resolver logic is duplicated into JS). A small `design-suite`/frontend JS helper `bhResolveBindings(bindings, ctx)` calls it and patches only the bound DOM nodes (identified by `data-placement-id` + attr key), so an element can refresh its bound value on an interval, on an event, or on demand **without a full page reload**. This directly answers "flexible at runtime" while keeping the security/sanitization boundary server-side. Elements opt in via a manifest flag (e.g. `'live' => true` or a per-attr `'refresh'` hint) so most elements stay static (cheap) and only genuinely-live ones poll/subscribe.
- **Output formatters (composable step 1).** Extend the binding descriptor with an optional `"format"` key naming a **registered formatter**: `{"bind": {"source": "...", "args": {...}, "format": "compact_number"}}`. Formatters register exactly like sources — `BH_Element_Data::register_formatter($slug, callable)`, zero-central, `class_exists`-guarded — and run on the resolved value before return (server) or in the JS helper (client, mirroring a small allowlisted set like `compact_number`/`relative_time`/`currency`). This is the "binding output transformed by a formatter" ask, added without touching `resolve()`'s core failure ladder (a formatter that throws degrades to the unformatted value, then to the literal — one more rung on the existing ladder).

**v2 (next, still incremental): chained/composable bindings.**

- Allow a binding's `args` values to reference another binding on the same placement by name: `{"bind": {"source": "...", "args": {"user": "@primary.value"}}}` where `@primary` names a sibling binding resolved first. Resolution becomes a small dependency-ordered pass (topological over `@`-refs within one placement, cycle-guarded → fall back to default on a cycle). This is genuinely more surface area than v1, which is why it is deferred — but the descriptor shape is designed for it now (`@name` refs are a strict superset of today's `context.*` tokens; `resolve_token()` gains one more prefix branch).

**v3 (last, most open-ended): client-state binding.**

- Extend the token vocabulary beyond `context.*` to `state.*` (live client-side state: a form input value, a selected tab, an interaction flag) and `sibling.*`/`parent.*` (another element's current resolved/DOM state). This is *only* meaningful client-side (the server has no `state.*`), so it rides entirely on v1's JS resolution helper: `bhResolveBindings` gains a client-state provider that reads registered client-state sources (form fields, a small reactive store keyed by placement id). This is the "bind to live client-side state, not just server data" ask, and it is deliberately last because it is the least-specified and the only part that needs a real client-side reactive contract rather than a re-fetch.

### 3.3 Proposed API contract (extends, never breaks, existing signatures)

```php
// class-element-data.php — additions, all backward-compatible:

// v1
public static function register_formatter($slug, callable $fn);      // zero-central, mirrors register_source()
public static function registered_formatters();                       // for the inspector's format dropdown
// resolve() gains: after a successful source resolve, if $binding['format'] names a
// registered formatter, run it; formatter failure = one more rung to the existing
// literal/default fallback ladder. No signature change to resolve().

// REST (class-element.php) — one new thin route:
// POST ous/v1/elements/resolve  -> wraps BH_Element_Data::resolve() for a batch of
//   bindings + a ctx; permission: bhcore_design_site OR the element's own surface
//   read policy (a live frontend element resolving its OWN bound value is a read,
//   gate accordingly — see §5 for the front-end boundary this must respect).

// v2 (descriptor only, resolver gains @-ref pass):
//   args values of form "@<bindingName>.value" resolve to a sibling binding first.

// v3 (client-only, JS helper contract):
//   window.bhResolveBindings(bindings, ctx) -> Promise<values>  (calls the REST route)
//   plus a client-state provider registry for state.*/sibling.*/parent.* tokens.
```

**Justification for this shape over a full client-side resolver rewrite:** the user said "at runtime," which is satisfied by re-resolution, not by moving resolution logic to the client. Keeping `BH_Element_Data::resolve()` the single authority for *what a source returns and how it is sanitized* preserves the security boundary the class was explicitly built around ("no PHP callable is ever persisted... deliberate security/portability boundary" — its own docblock). The JS helper is a *transport + DOM-patch + client-state* layer, not a second resolver. This is the same discipline `LMS-AUTHORING-DESIGN-PLAN.md` applied to the table view (a second projection over one storage contract, not a second store).

Lower priority than the core groups — the `config.style`/`htmlAttrs` namespaced-key convention and `scoped_inline_style()` formatter already accommodate them, so this is additive UI over a settled storage/render seam. **Testable:** an element given `display:flex` + `position.z-index` + a custom `data-role` attribute renders correctly and round-trips through save/reload.
6. **Data-binding v2/v3 (chained bindings, client-state).** The most open-ended; ship only after 1-5 are stable. Named, seams present (`@`-ref token branch, `state.*` provider), not built before their consumers exist.

**The just-built CRM kanban board — leave as-is, with a note.** The project tracker (class-projects.php, kanban-board.js) predates this redesign and already consumes the correct bridges (`ous/v1/elements/placements/*`, `BH_Content` sub-cards, `BH_Element_Prefab`). It should **not** be forcibly migrated onto the `BH_Design_Suite` shell in this pass: its board UI is an intentionally *bespoke* presentation layer (per its own docblock), and rebuilding it as a generic scene-tree surface would regress a working, purpose-fit UI for no user-visible gain. It benefits automatically from Phase 2 (its `bh/sticky-card` placements can carry `config.style` overrides) and Phase 5 (roll-up labels could become live) with zero board-code changes. Migration of the board *into* the unified tree is explicitly a future, optional call — flagged, not scheduled, matching the ecosystem's "one real example, not every consumer at once" discipline.
