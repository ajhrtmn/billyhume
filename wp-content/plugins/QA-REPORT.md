## Original report (2026-07-07, earlier session)

Session date: 2026-07-07. Scope: the six-plugin bundle in `own-ur-shit-ecosystem-v20.zip` (`own-ur-shit` core + `bh-streaming`, `bh-contest`, `bh-courses`, `bh-crm`, `bh-registry`, `bh-monetization-woo`), against the handoff brief's request for a full live QA pass plus a DRY/SOLID refactor.

## 1. The environment constraint — read this first

The brief asked for a real WordPress + MySQL install (Local/wp-env/Docker), activation on fresh and pre-populated sites, and exercising every admin screen, front-end flow, the Debug Tools seed/reset actions, and the Test Runner's "Run all tests" button.

**None of that was possible in this session.** The available execution environment (a sandboxed Linux shell) has no PHP, no MySQL, no Docker, and no root access — `apt-get install php-cli` fails with a permission-denied dpkg lock, and `sudo` is architecturally disabled ("no new privileges" flag set). Outbound network access is a default-deny egress proxy that blocks everything, including plain domains tested purely as a sanity check (`google.com`, `github.com`, `wordpress.org`, `packagist.org` all returned `403 blocked-by-allowlist`) — so there was no path to downloading a static PHP binary or using a WASM PHP interpreter either. A leftover `/tmp/phpinstall.log` in the sandbox showed an earlier, unrelated session hit the identical wall. A remote-sandbox probe was also tried specifically to check for a less-restricted execution environment, and came back with the same three independent blockers (no root, no package manager access, no egress).

Given that, this pass is **static analysis only** — careful reading, automated lexical checking, and reasoning, not execution. Every fix below is "the logic is now sound," not "confirmed working against a live database." Section 5 lists everything that still needs a real WordPress+MySQL environment to actually verify.

One packaging note: `bh-contest` was not present as a top-level plugin folder in the delivered zip — it's bundled inside `own-ur-shit/bundled/bh-contest.zip` (the mechanism `OUS_Registry`'s `bundled_zip` entries use for one-click install from the dashboard). It was extracted from there for this review and is included as a full top-level plugin in the refactored zip alongside the other five.

## 2. Method

1. Built a proper PHP-lexing static checker (Python) that correctly tokenizes single/double-quoted strings, heredoc/nowdoc, comments, and `<?php`/`?>` mode transitions before counting brace/paren/bracket balance — replacing the naive counter a previous session used, which had known false positives on regex character classes and CSS selectors, and would have produced a false positive on this codebase's own raw-HTML-between-`?>`-tags style if left unfixed (caught and fixed mid-session; see below). Ran it across all 100+ PHP files. **Result: zero unbalanced-brace or unterminated-string findings**, including the specific "unescaped apostrophe silently ends a long single-quoted string" bug class flagged as a known recurring issue.
2. Grep-hunted the other four known bug-class patterns from the handoff brief directly:
   - Chicken-and-egg plugin registration (`ous_registered_plugins` self-registration never firing for an inactive plugin) — confirmed already fixed; `OUS_Registry::DEFAULTS` hardcodes all six peer plugins plus WooCommerce and Advanced Media Offloader, with each plugin's own filter-based self-registration left in place as a harmless once-active no-op.
   - `class_exists()` checks at file-parse time vs. call time — checked every occurrence across the codebase; all are correctly placed inside `plugins_loaded`/`init` hook callbacks, never at top-level file scope. `BHCORE_LOADED` is defined at the top of `own-ur-shit.php`, but every peer plugin only ever checks it inside a `plugins_loaded` callback, which is guaranteed to run after every active plugin's main file has already executed — so the "own-ur-shit loads alphabetically last" concern doesn't actually bite here. No bug found.
   - Screen-ID substring matching for `BHY_UI`'s shared CSS/JS — confirmed already broadened from a strict `'bh-'` match to bare `'bh'`/`'ous'`/`'own-ur-shit'`, matching the documented real screen-ID shapes (`edit-bh_course`, `bhs_track_page_bhm-settings`, etc.). Could not verify this against actually-rendered screen IDs (needs a live install — see Section 5).
   - Data migrations (`BHS_Activator::rename_post_types_to_prefixed()`, `BHM_Activator::migrate_purchase_object_type_meta()`) — read in full; both are idempotent, correctly parameterized, version-gated, and run on both `activate()` and `maybe_upgrade()`. Logic reads as correct; could not test against a real database with pre-existing old-format rows (see Section 5).
3. Spawned seven parallel deep-read reviews (one per plugin, one for the core), each reading every file in full and cross-checking every cross-class/cross-plugin method call, nonce/capability check, `$wpdb->prepare()` usage, and output-escaping site. Findings consolidated and fixed below.
4. Fixed every confirmed bug, then did a scoped DRY/SOLID/encapsulation pass informed by what the reviews actually found (see Section 4) rather than refactoring speculatively.

## 3. Bugs found and fixed

| # | Plugin | File | Bug | Fix |
|---|---|---|---|---|
| 1 | bh-streaming | `class-player.php` | Unconditional `BHY_Style::inline_css()` call with no `class_exists()` guard — the one exception to this ecosystem's own convention. Would fatal every `[bh_streaming]` page if `own-ur-shit` weren't loaded at that point. | Guarded with `class_exists('BHY_Style')`. |
| 2 | bh-streaming | `class-recommendations.php` | `/tracks/{id}/related` resolved audio URLs directly from postmeta instead of `BHS_API::track_payload()`: shipped a real playable URL for monetization-locked tracks (paywall bypass), and silently dropped every externally-aggregated track (only checked `_bhs_audio_id`, never `_bhs_external_audio_url`). | Routed through `track_payload()`, which applies the `bhs_track_access_allowed` filter and the correct local-or-external URL resolution. |
| 3 | bh-courses | `class-progress.php` | `mark_step_complete()` bound `null` score/passed values through `%s`/`%d` `$wpdb->prepare()` placeholders, which cast `null` to `''`/`0` — there's no NULL passthrough. Since `is_step_complete()` explicitly tests `=== null` to detect "non-quiz step, done on write," **every plain text/image step in every course permanently read as incomplete**, breaking course-percent, resume-progress, and completion detection. | Build the SQL with a literal `NULL` when the value is null, instead of a bound placeholder. |
| 4 | bh-courses | `class-progress.php` | An already-passed quiz step could be resubmitted and rescored, and a failing resubmission would overwrite the passing result server-side (front end disables the resubmit button, but that's not a server-side guarantee). | Short-circuits and returns the existing passing result once `$already_passed` is true. |
| 5 | bh-courses | `class-gate.php` | Drip-content date logic mixed `current_time('timestamp')` (site-timezone-adjusted) with raw `strtotime()` of stored strings (PHP-default-timezone) — a real boundary-condition bug on any site whose WP timezone differs from the server's PHP default. | Resolve the admin-picked calendar date against `wp_timezone()` explicitly; compare enrollment timestamps against raw UTC `time()` consistently. |
| 6 | bh-contest | `class-api.php` | `vote()` checked contest ownership but never `post_status === 'publish'` — a user who knew/guessed a pending or rejected submission's ID could burn a limited vote on a track that could never appear in results. | Added the publish-status check alongside the existing type/contest check. |
| 7 | own-ur-shit | `class-public-profile.php` | Avatar/banner upload handler had no capability check before `media_handle_upload()` (which performs none of its own) — since self-registered accounts are plain subscribers without `upload_files` by default, this was a real "any logged-in low-privilege user can write into the media library" gap. | Scoped, request-duration-only `upload_files` grant around just this call (`user_has_cap` filter, added then immediately removed) rather than a hard block that would have broken the intended subscriber-facing feature. |
| 8 | own-ur-shit | `class-registry.php` | `status()`/`version()` indexed the registry array directly instead of `isset()`-guarding, unlike `OUS_ActivationManager` elsewhere in the same plugin. Not exploitable today (every call site passes a known-good key) but inconsistent and would throw a PHP 8 warning on a future stale key. | Added the same `isset()` guard, returns `'missing'`/`''` gracefully. |
| 9 | bh-streaming | `class-feeds.php` | A feed-fetch network failure (timeout/DNS/host-down) was indistinguishable from a genuinely malformed feed — both showed "didn't look like an open podcast/audio feed," pointing admins at the wrong problem. | Tracked as a distinct `_bhs_feed_fetch_failed` meta flag with its own message; the "malformed feed" message now only shows for an actual validation failure. |

All nine fixes were verified against the lexical checker afterward (still zero unbalanced-brace/unterminated-string findings across all 102 files) and re-read for internal consistency, but **none were executed**.

### Notable things checked carefully and found correct (worth recording so they aren't re-litigated)
- `BHS_Activator::rename_post_types_to_prefixed()` and `BHM_Activator::migrate_purchase_object_type_meta()` — idempotent, correctly parameterized, version-gated on both `activate()` and `maybe_upgrade()`.
- `WC()`/order/product API usage throughout `bh-monetization-woo` matches real WooCommerce API shape; the wallet ledger's atomic `UPDATE ... WHERE balance_cents >= %d` and `INSERT ... ON DUPLICATE KEY UPDATE` patterns correctly avoid the read-then-write race the codebase's own comments flag as a risk.
- `BHI_TwoFactor`'s TOTP/base32 implementation and both its PHPUnit and Test-Runner suites are internally consistent with each other and with RFC 6238.
- `BHC_Steps::score_quiz()` and `save()` sanitization match their PHPUnit test assertions exactly (a real risk category, since those tests were also never executed).
- Nonce verification, capability checks, `$wpdb->prepare()` usage, and output escaping were checked file-by-file across all seven plugins; aside from the two items above (#7, #8), no other gaps were found.

## 4. DRY/SOLID/encapsulation refactor — what was done and what was deliberately left alone

The brief (and a follow-up question mid-session) asked specifically for good abstraction and encapsulation, not just deduplication. Given no live environment to verify a broad rewrite doesn't break a call site, this pass scoped to changes that were either mechanically safe (identical behavior, just relocated) or isolated enough to review with full confidence by reading:

- **`bh-monetization-woo`'s `class-products.php` split (single-responsibility).** It was doing five genuinely distinct things: WooCommerce product sync, streaming-gate hooks, pay-per-play billing, refund/fraud-pattern detection, and entitlement granting. Refund/fraud-pattern detection — the most self-contained of the five, confirmed by tracing every call site — was extracted into a new `BHM_Fraud` class (`includes/class-fraud.php`), with its own `fingerprint_for()` helper now properly `private` and encapsulated inside it rather than living as a private method on an unrelated god-class. `class-products.php` and `class-crm-integration.php`'s external read of `refund_count_recent()` were updated to call the new class. The other four responsibility groups are more tightly coupled to each other (gating and billing both depend on `is_non_catalog_track()`) and were left as one class rather than force-split blind.
- **`bh-registry` data-access encapsulation.** The plugin had no data-access layer at all for its own `bhr_links` table — 14 call sites across 6 files each independently reached into `$wpdb->prefix . 'bhr_links'` and hand-wrote SQL, including three byte-identical "fetch one link by id" queries. Added a small `BHR_Links` class (`table()`, `find()`) and migrated those three exact-duplicate call sites to it. The more varied queries (filtered lists, bulk deletes) were deliberately left where they are rather than forcing everything through a heavier repository abstraction the plugin doesn't need yet.
- **`OUS_Debug::register()` convenience wrapper.** Every one of the six `ous_debug_tools` registrants hand-rolls an identically-shaped `add_filter` closure. Added an optional wrapper method to the core (purely additive) for future registrants to use; deliberately did not retrofit the six existing, working call sites, since that's a mechanical-looking change across six files with no way to confirm none of them broke.

### Identified but deliberately not touched this pass (flagged for a follow-up with a real test environment)
- Repeated `class_exists('WooCommerce')` guards at the top of nearly every public method in `bh-monetization-woo`, instead of one central check.
- Cents-to-dollar-string formatting (`number_format($x / 100, 2)`) repeated ~10+ times across `bh-monetization-woo`'s admin/frontend/CRM classes with no shared helper.
- Badge/status-color lookup tables hand-rolled independently in several core classes (`class-debug-log.php`, `class-api-docs.php`, `class-jobs.php`) instead of reusing `BHY_UI`'s existing `.bhy-badge-*` classes — these appear to predate the shared design system and were never migrated.
- The `render_debug_section()`/`handle_debug_action()`/`reset_debug()` triad and the tag-based "seed test data, wipe by tag" pattern are structurally duplicated across every plugin's own `class-debug.php` — the core's `OUS_Debug` could plausibly offer a generic seed/wipe helper, but retrofitting six plugins' working seed logic without being able to run their "Reset" buttons afterward was judged too risky for this pass.
- The "external HTTP fetch with graceful degradation" pattern in `bh-streaming`'s feed sync and `bh-registry`'s domain/ActivityPub verification is real but only partially overlapping in shape (see the per-plugin review notes) — a shared low-level "guarded `wp_remote_get`" helper is plausible, a shared high-level policy helper is not, given the two plugins' different trust models (admin-curated aggregation vs. voluntary self-serve registry).

None of the above are bugs — they're duplication/encapsulation debt, explicitly deferred rather than silently ignored.
