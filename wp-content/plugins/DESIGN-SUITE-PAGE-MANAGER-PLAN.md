# Design Suite as the page/template manager — plan

Companion to `DESIGN-SUITE-UNIFICATION-PLAN.md` (the "no special-cased pages" architecture this whole doc extends outward to ordinary WP pages/posts, the one class of page that architecture has never touched yet). AJ's own framing, verbatim, that scopes this doc: *"lets make sure its good and airtight so we dont miss anything"* on top of *"I basically will want [the Design Suite] to become the whole builder/page and post manager (at least for the templates and such - like HubSpot kinda does)... I dont want to over accommodate wordpress's conventions if they suck... but make sure the UX for it is clear and intuitive to make them work seamlessly together."*

**Status as of 2026-07-12: DESIGN ONLY, NOT BUILT.** Written before any code, per this session's own standing convention for a pivot this size (see `PROJECT-TRACKER-TRACKIT-PARITY-PLAN.md`'s identical posture).

## 0. The one decision that shapes everything below

AJ picked explicitly between two options offered:

- **Option A (rejected):** the Design Suite fully replaces `wp_posts` for pages — a "page" is just a Site-tree node, no post row at all. True end-to-end ownership, HubSpot's actual architecture, but means reimplementing routing, slugs, revisions, trash, search indexing, and every third-party plugin (Yoast, sitemaps, related-posts, etc.) that assumes a real post exists.
- **Option B (CHOSEN):** a WP post/page still exists — real `wp_posts` row, real permalink, real `post_type`, real revisions/trash/search/Yoast/sitemap support, all untouched — but its RENDERED BODY is 100% node-tree. `the_content` is not authored in the block/classic editor at all for an opted-in post; it's generated at render time by walking that post's own `BH_Element` slot, the exact same `render_slot()` call every other surface in this ecosystem already uses.

Everything below assumes B. The theme's own header/footer/sidebar/page-template chrome around `the_content` is completely untouched — this only replaces what goes INSIDE the content area, never the surrounding page shell.

**Option A is not dead, just not now — AJ's own words, recorded so this isn't re-litigated from scratch later:** *"there may come a time where this is the right call though... especially with the issues we've had with the rewrites for different links and such. If what is there can't be abstracted against and made to work, then we should usually consider doing our own, so it's portable for a WP-free future."* This session has real, on-the-record scar tissue with exactly the class of problem Option A would sidestep: the Portal rewrite-rule investigation, the standalone-page registration/`add_submenu_page(null, ...)` saga, and the general pattern of WP's own routing/menu system fighting this ecosystem's own architecture rather than getting out of its way. Option B is the right call FOR NOW because it's cheap (keeps permalinks/revisions/Yoast/sitemaps for free) — but that calculus flips the moment `wp_posts`/rewrite rules become a wall this ecosystem keeps hitting rather than a convenience it's borrowing. If Option B's Phase 1-5 build-out (§5) runs into a rewrite/routing wall serious enough to need its own investigation (matching the pattern of prior sessions' rewrite-rule work), that is the trigger to revisit Option A directly, not push further into working around WP's routing. Worth designing Option B's own internals (§1's `bh_page` surface, `render_slot()` call, the picker in §3) so they're NOT WP-specific where avoidable — i.e. so a future Option A migration could reuse the exact same node-tree/rendering layer and only have to replace the routing/storage shell around it, not rebuild the content model too.

## 1. Data model — what's actually new

Almost nothing. This is intentionally a thin generalization of a pattern already proven twice this session (CRM profile pages, LMS lesson pages), not a new subsystem:

- A new surface, registered the same `bh_element_surfaces` way every other surface is: `bh_page`, one `root` slot, `'context' => ['type' => 'post', 'param' => 'post_id']` — `surface_context_id` is just the WP post ID, exactly like the LMS lesson surface already does. No new table.
- One new postmeta key, `_bh_design_suite_managed` (bool-ish `'1'`/absent), the single source of truth for "is this specific post's content owned by the Design Suite or by WP's native editor." Per-POST, not per-post-type — a judgment call, explained in §4.
- A `the_content` filter (mirroring `bh-courses`' own existing `bh_lesson` filter almost exactly — see `bh-courses.php`'s bootstrap) that, when the current post has `_bh_design_suite_managed`, REPLACES `$content` entirely with `BH_Element::render_slot('bh_page', $post_id, 'root', $ctx)` rather than appending to it (LMS's own integration appends, deliberately, since a lesson's step content is a separate, pre-existing system this doc's own earlier pass was careful not to touch — a page has no such pre-existing system to preserve alongside, so full replacement is correct here, not a smaller append).

That's the entire runtime piece. Everything else in this doc is UX — making that one filter and one postmeta flag discoverable, safe, and reversible from inside wp-admin, which is where "airtight" actually has to be earned.

## 2. The wp-admin post-edit-screen seam (the part most likely to be gotten wrong)

This is the single highest-risk UX surface in the whole plan: what does an author SEE on `post.php?action=edit` for a page that may or may not be Design-Suite-managed, and what happens the moment they toggle it.

**New metabox, "Design Suite," registered on every opted-in post type (see §4 for which), replacing the native content editor's visible role rather than sitting alongside it silently:**

- **Not yet opted in (default for an existing page):** the box shows a single toggle, "Build this page with Design Suite," off by default, plus one sentence of what it does. The native block/classic editor stays exactly as it is — fully live, fully in control — until the toggle is switched on. Nothing changes for a page nobody touches.
- **Opting in:** the moment the toggle is switched on and the post is saved, `_bh_design_suite_managed` is set, AND — the non-negotiable part — the post's EXISTING `post_content` is never silently discarded. It is auto-wrapped as a single starting `bh/note` placement in the new `root` slot (one `BH_Element::save_placement()` call, config `['text' => $existing_post_content]`), so opting in is a pure superset: whatever was there is still there, now as the first (and only, until edited) node in the tree, immediately editable/deletable/restructurable like any other node. **This is the single most important non-destructive guarantee in this whole doc — flagged here explicitly so it's never "optimized away" in a later pass.**
- **Opted in:** the native content editor is HIDDEN, not left present-but-inert. A present-but-ignored editor is a guaranteed support-ticket generator ("I edited the page and nothing changed") — hiding it and replacing it with a clear, unambiguous panel is the correct call, made explicitly here rather than left as an implementation-time guess. In its place: a short "This page's content is managed by the Design Suite" notice plus one big button, **"Open in Design Suite →"**, deep-linking straight into the Design Suite with this exact page pre-selected and expanded in the tree (the picker/deep-link mechanics are §3). The title field, publish box, categories/tags, featured image, SEO plugin metaboxes, and everything else on the edit screen are completely untouched — only the content editor itself is swapped out.
- **Opting back out:** the toggle can be switched off again at any time. This is explicitly NOT fully seamless, and the doc says so rather than promising something that isn't true: `post_content` is whatever it was frozen at the moment of opt-in (or empty, if the page was Design-Suite-native from creation — see §4's default-on discussion) — turning the toggle off does NOT reverse-render the current node tree back into `post_content`. The node-tree placements themselves are never deleted on opt-out (harmless orphaned data, cheap to keep, expensive/lossy to reconstruct) — opting back in later shows the exact same tree exactly as it was left. This is a real, honestly-disclosed rough edge: round-tripping opt-in/opt-out is safe and non-destructive, but not lossless-and-symmetric. Flagged as a judgment call in §4 rather than silently shipped.

## 3. The Design Suite tree UX (closing the exact gap AJ's own screenshot flagged)

Today, a per-entity surface (CRM profile, LMS lesson) is reachable in the tree only via a raw numeric "Context ID (advanced)" spinner on the Slot node's inspector — functional but exactly the kind of form-field UX this whole architecture was built to get away from, and it does not scale to "pick one of potentially hundreds of pages." This plan requires a real fix, not a bigger version of the same spinner:

- **New "Pages" (and, if Posts opts in — §4 — a separate "Posts") group under Site in the tree**, sibling to the existing CRM/Dashboard/etc. groups, populated by a real search-as-you-type picker (a small REST route, `GET ous/v1/elements/context-picker/{surface}` — new, since today's `GET .../elements/surfaces` genuinely carries no entity metadata to build a picker from, an honestly-disclosed gap the earlier per-entity-surface pass already flagged and left for "a future pass" — this is that pass) returning `{id, label}` pairs (post title + a small "managed"/"native" badge) for every post of an opted-in type. Typing filters the list; clicking a result expands that exact post's `root` slot right there in the tree, precisely the way clicking "CRM profile page" already does for a fixed context-0 entity today.
- **"+ New Page" action** at the top of that group: prompts for a title only (slug auto-generates from it, exactly like WP's own native "Add New" flow), creates a real `wp_posts` draft via `wp_insert_post()`, sets `_bh_design_suite_managed` immediately (a page created this way starts Design-Suite-native, no legacy content to preserve), and drops straight into that page's now-empty `root` slot in the tree — zero trip through wp-admin's own post-new.php screen required for the common case. Publishing still happens through the normal WP publish flow (see §5 — deliberately NOT reimplemented inside the Design Suite this pass).
- Every context-picker result's "managed" badge doubles as the honest visibility this whole plan needs: an author browsing the tree can tell at a glance which pages are Design-Suite-owned and which are still native WP content, rather than guessing.

## 4. Judgment calls, flagged explicitly rather than silently decided

- **Default-on for new Pages vs. staying opt-in-only, even for brand-new ones.** AJ's own framing ("the whole builder/page and post manager... at least for the templates and such") leans toward Design Suite being the DEFAULT going forward, not a checkbox someone has to remember to flip. Recommendation: **default new posts of type `page` to Design-Suite-managed at creation** (the "+ New Page" flow above already does this) while leaving the toggle available to opt a specific page back to native if a specific page genuinely needs something the tree can't do yet (a raw embed, a third-party shortcode-heavy legacy page, etc.). **Posts** (the blog `post` type) are recommended to stay fully opt-in, default OFF — a blog post is much more likely to be authored in a fast, native, keyboard-driven editing flow (this is the same instinct that kept LMS lesson-step content on `BH_Studio`/`BH_Content` rather than folding it into this same mechanism) and forcing every blog post through node-by-node construction would be a real regression for that specific authoring style, not an improvement. This is a real, overridable call — flagged for AJ's own sign-off before Phase 2 ships, not assumed silently.
- **Which post types register the metabox/toggle at all.** Recommendation: `page` unconditionally; `post` opt-in per the point above; custom post types from other plugins (bh_course, bh_lesson, bh_contest entries, etc.) explicitly EXCLUDED from this generic mechanism — those already have (or, for LMS lessons, just got) their own purpose-built `BH_Element` surfaces with their own additive/scoped semantics, and folding them into the generic `bh_page` surface too would create two competing systems fighting over the same post. `bh_page` is specifically for WP's own built-in `page`/`post` types; a plugin-owned CPT stays on its own dedicated surface.
- **Opt-out losing the live tree/post_content sync**, per §2's last bullet — recommended to ship as-is (non-destructive but not perfectly symmetric) rather than delay this whole plan on building a full node-tree-to-`post_content`-HTML serializer, which is real, non-trivial work with its own edge cases (what does a live-bound stat-card even mean once frozen as static `post_content` HTML?) that doesn't obviously pay for itself given opt-out is expected to be rare once a page has been built in the tree.
- **Cleanup on post trash/delete.** Recommended: hook `before_delete_post` (real delete only, not trash — trash is recoverable, so placements should survive a trash/restore cycle same as the post itself does) to delete that post's `bh_page`/`root` placements via a new `BH_Element::delete_context($surface, $context_id)` helper (doesn't exist yet — small, mechanical addition). Without this, every genuinely-deleted page leaves orphaned rows in `bhcore_element_placements` forever — cheap individually, real hygiene debt at scale. Flagged as its own phase (§5, Phase 5) rather than bundled into Phase 1, since it's non-urgent cleanup, not a correctness requirement for anything else in this doc to work.
- **Live preview / "Preview changes" while editing in the Design Suite.** WP's native "Preview" button on the edit screen already works with zero changes needed here — it's still a real post, still resolves through the normal permalink/template pipeline, and the `the_content` filter re-renders the CURRENT saved tree state on every request, same as any other dynamic content. The one honest gap: the Design Suite's own canvas mirror (the "(live)" story added in the previous pass, 3.4.48) shows the tree's state as of the last "Save all changes" click, not unsaved in-progress edits — matching the existing save-then-preview model every other surface already has, not a new inconsistency introduced by this plan.

## 5. Phased build plan

- **Phase 1 — Data model + render path only, no admin UX.** `bh_page` surface registration, `_bh_design_suite_managed` postmeta convention, the `the_content` replacement filter. Testable directly (set the postmeta, add a placement via the existing REST/tree UI against surface `bh_page`/context = a real post ID, confirm the front end renders it) before any wp-admin screen work exists.
- **Phase 2 — Post-edit-screen metabox.** The toggle, the non-destructive `post_content`-to-starting-node wrap on opt-in, hiding the native editor + "Open in Design Suite →" deep link on opt-in.
- **Phase 3 — Tree UX: the context picker + "+ New Page."** The new `GET ous/v1/elements/context-picker/{surface}` route, the "Pages"/"Posts" tree groups, the managed/native badge, in-tree page creation.
- **Phase 4 — Default-on for new Pages** (pending AJ's sign-off on §4's judgment call) — flips `page` creation (both WP's native "Add New" and the Design Suite's own "+ New Page") to Design-Suite-managed by default.
- **Phase 5 — Cleanup hook** (`before_delete_post` -> `delete_context()`) — hygiene, non-urgent, safe to trail the rest.

Recommended build order: 1, 2, 3, 4, 5 — strictly ascending risk/visibility, matching every other phased doc in this repo. Each phase gets its own live-verified pass (screenshot-confirmed opt-in round trip, tree pick, front-end render) before starting the next, per this session's own standing lesson about not shipping multiple speculative UI passes unseen.

## 6. Explicitly NOT in scope

- Rebuilding routing, permalinks, revisions, trash, or search indexing — all stay 100% native WP, since Option B was chosen specifically to keep these for free.
- Any change to WP block templates / `page-*.php` theme template selection — the surrounding chrome outside `the_content` is untouched.
- Reconstructing `post_content` HTML from a node tree on opt-out (§4).
- Importing content from third-party page builders (Elementor, Divi, etc.) into node-tree form — a real, separate feature if ever wanted, not implied by anything here.
- Multisite-specific considerations — not evaluated, this install isn't multisite.
- Folding plugin-owned CPTs (bh_course, bh_lesson, bh_contest, etc.) into `bh_page` — explicitly excluded per §4; those keep their own dedicated surfaces.
