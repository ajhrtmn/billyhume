# ROADMAP: Discoverability — SEO, AI-answer-engine visibility, federated distribution

Companion to VISION.md's "Marketing tools — currently nothing exists here at all" gap (already flagged there) and the round-three federation pillar. Scopes the actual plan for "let the artist focus on creating, automate getting the word out" (AJ, 2026-07-17) — planning and one real first slice, not the whole ambition built at once.

## 0. Grounded in what's actually there today

Confirmed by checking, not assumed:
- **Zero meta tags, Open Graph, Twitter Card, or structured data (JSON-LD/schema.org) anywhere in this codebase** — a full grep across every plugin for `og:title`, `schema.org`, `application/ld+json` returns nothing.
- **WordPress core's own native XML sitemap (`/wp-sitemap.xml`, built into WP since 5.5) is already live and already includes `bh_course` and `bh_lesson`** — confirmed by fetching it directly. This is a real, working piece of infrastructure this ecosystem gets for free, not a gap to build from scratch. It does NOT currently include `bh_contest`/`bh_submission` (both registered `'public' => false`, so WP correctly excludes them — right call for submissions, worth a second look for whether a published, results-revealed CONTEST itself should be publicly indexable) or bh-streaming's `bhs_track`/`bhs_release` (needs checking whether those are registered public and simply excluded some other way, or genuinely not indexable yet).
- **RSS/Podcasting-2.0 export already exists** (bh-streaming's own catalog feed, per its README) — this is already a real, working piece of the "automate getting the word out" ask, just not framed as one yet.
- **ActivityPub federation is already scoped** (VISION.md's round-three pillar, building on `bh-registry`) — the federated-distribution half of this ask is a planned, not-yet-built continuation of that same work, not a separate initiative.

## 1. Three distinct problems, worth naming separately

**Traditional SEO** (Google/Bing indexing, ranking) is the most well-understood of the three — meta descriptions, Open Graph/Twitter Card tags for link-preview quality, canonical URLs, and schema.org structured data (JSON-LD) so search engines understand what a page actually IS (a `MusicGroup`, a `MusicRecording`, a `Course`, an `Event` for a contest) rather than guessing from prose. This is well-trodden, low-risk, and the highest-confidence part of this whole roadmap.

**AI-answer-engine visibility** ("AI response authority," AJ's own phrasing) — being the source an LLM-backed answer engine (ChatGPT, Perplexity, Google's AI Overviews) actually cites when someone asks about an artist or their work. This is genuinely newer and less standardized than classic SEO, worth being honest about: the two concrete, real mechanisms as of mid-2026 are (a) the SAME structured data/clean-semantic-HTML that helps classic SEO also helps an LLM's retrieval step parse a page correctly — there's no separate "AEO markup" standard distinct from good schema.org usage — and (b) `llms.txt`, an emerging (not yet formally standardized, but genuinely adopted by a real, growing set of sites and crawlers) plain-Markdown file at a site's root that gives an LLM crawler a clean, curated summary of what's on the site and links to the highest-value pages, the same spirit as `robots.txt`/`sitemap.xml` but written for a language model's consumption pattern (prose + links) instead of a search index's. Treat `llms.txt` as a cheap, low-risk bet worth taking, not a guaranteed-effective silver bullet — verify against current, live documentation before scoping further, same discipline VISION.md's own social-platform research sections already apply.

**Federated/open distribution** — RSS/Podcasting-2.0 (already real) and ActivityPub (already scoped, not built) are the two concrete mechanisms; both let an artist's own content surface in OTHER people's feeds/apps without needing a platform's algorithm to decide to show it. This is the piece most aligned with the ecosystem's whole anti-middleman philosophy — visibility that comes from open protocol adoption, not from paying or performing for a ranking algorithm.

## 2. The automation framing — what "let the artist focus on creating" actually means here

The concrete promise: an artist publishes a track, a course, a contest — and the RIGHT metadata, structured data, sitemap inclusion, and feed entries happen automatically, with zero separate "now go do your SEO" step. This is an argument for building one shared mechanism every content type plugs into (matching the established `ous_debug_tools`/`bhi_portal_panels`/`bhcore_metrics_widgets` pattern), not a per-plugin bolt-on each content type would otherwise have to remember to implement separately.

## 3. Proposed shared mechanism: `BH_SEO` (own-ur-shit core)

- **`BH_SEO::render_head_tags()`** — hooked on `wp_head`, reads a `bh_seo_page_data` filter (same zero-central-registration shape as everything else) that any plugin populates for the CURRENT page: title, description, canonical URL, OG image, OG type, and a structured-data array. Renders the actual `<meta>`/OG/Twitter Card tags plus a single `<script type="application/ld+json">` block from the structured-data array — one shared renderer, so every plugin's output is consistently correct instead of each hand-rolling its own tag soup.
- **Per-content-type schema.org shapes to design deliberately, not guess at**: a track/release is `MusicRecording`/`MusicAlbum`; an artist's public profile is `Person` or `MusicGroup`; a course is `Course` (schema.org has a real, specific `Course`/`CourseInstance` shape worth using correctly, not just a generic `CreativeWork`); a contest is plausibly `Event` (for the live reveal) or `CreativeWork` for the archive view — worth a real short design pass matching each content type to its correct schema.org type rather than reusing one generic shape everywhere.
- **`llms.txt`** — a small, mostly-static endpoint (own-ur-shit core, `template_redirect` on the `/llms.txt` request) assembled from the same site-wide content this ecosystem already knows about (site name/bio, links to the streaming catalog, course catalog, registry profile if public) — genuinely cheap once `BH_SEO`'s page-data filter exists, since it's mostly a different rendering of the same underlying data.
- **Sitemap gap-filling**: audit which of this ecosystem's own public post types (`bhs_track`, `bhs_release`, published `bh_contest`) are NOT currently appearing in WP core's native sitemap and fix the specific `'public'`/`show_in_rest` registration flags causing that, rather than building a parallel custom sitemap system — WP core's own mechanism already works, it just needs every post type that SHOULD be public to actually be registered that way.

## 4. What's NOT scoped yet, flagged honestly

- No code written yet for any of the above except this plan.
- Federated distribution itself (ActivityPub) is covered by VISION.md's existing round-three pillar, not re-scoped here — this doc's "federated" section is about USING what already exists (RSS) and what's already planned (ActivityPub), not designing federation from scratch a second time.
- `llms.txt`'s real-world effectiveness is genuinely unproven/evolving — worth checking current adoption/guidance again whenever this gets picked up, not treated as settled by this doc.
- Paid/algorithmic discovery (actual ad spend, influencer/platform partnerships) is deliberately out of scope — matches VISION.md's own "no ads, ever" stance; this roadmap is about EARNED and OPEN-PROTOCOL discoverability only, never paid placement.
