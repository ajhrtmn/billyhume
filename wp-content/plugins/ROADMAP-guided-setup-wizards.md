# ROADMAP: Guided Setup Wizards ("it just works")

Companion to VISION.md's "It just works" section — that doc states the principle; this doc scopes the actual framework and its first real implementation. Not built yet — concrete plan and technical grounding, ready to pick up.

## 0. Why this exists, stated plainly

Owning your infrastructure only beats renting it if the ownership part doesn't feel like a chore. A self-hosted CDN/storage setup that requires reading a provider's own API docs and hand-copying credentials between two dashboards will lose to "click Connect" every time, even though it's worse long-term. This framework exists to close that gap — not by hiding the complexity, but by walking a non-expert through it one plain-language question at a time, the way TurboTax turns a tax return into an interview instead of a form.

The trigger for scoping this now: **long, high-resolution video needs to not become a headache** (AJ, 2026-07-17) — a real, concrete requirement, not a hypothetical. That constraint shapes the whole plan below more than the general "wizard framework" ambition does, so it's addressed first and in the most technical depth.

## 1. The actual video-streaming problem, grounded (not assumed)

Confirmed via reading the code, not assumed:
- `bh-streaming`'s player (`class-player.php`) is deliberately minimal — no chapter markers, no overlay hooks, no adaptive-bitrate awareness. It plays whatever URL it's given via a plain HTML5 `<video>`/`<audio>` tag (or Howler for audio).
- `bh-courses`' video step (`class-steps.php`) is the same shape: `source: 'upload'` (a WordPress media-library attachment, served via `wp_get_attachment_url()`) or `source: 'url'` (an external embed/direct link).
- **Advanced Media Offloader** (third-party, already installed) offloads WordPress media-library files to S3/Cloudflare R2/DigitalOcean Spaces/Backblaze B2 — a bare settings page (bucket/region/keys), no wizard, no video-specific handling. It solves "don't fill up my own server's disk," nothing about streaming quality.

**What "just move the file to cheaper storage" does NOT solve for long, high-res video:**
- **No adaptive bitrate.** A single MP4 file, however it's hosted, is one fixed quality. Someone on a slow connection either waits a long time for the first frame or gets a player that stalls repeatedly — there's no automatic step-down to a lower-bitrate rendition the way YouTube/Netflix/Vimeo do it.
- **No automatic transcoding.** A creator uploading a 4K ProRes export gets exactly that file served to every viewer, regardless of their device or connection — nothing re-encodes it into something more deliverable.

**What a plain object-storage-plus-CDN setup (R2/S3/Azure Blob behind Cloudflare/CloudFront) DOES already solve, correctly, for free:**
- **HTTP range-request support** — S3-compatible storage and every major CDN support `Range:` headers natively, which is what makes seeking/scrubbing and fast start-of-playback work at all for a direct MP4. This is NOT an adaptive-bitrate feature, but it's the single most important thing for "video doesn't feel broken to use," and it costs nothing extra to get right — it just requires the storage/CDN layer to actually be configured (the thing this wizard exists to make easy).
- **Bandwidth cost and origin load moved off the artist's own server** — the actual "doesn't become a headache" part of the ask. A single popular long video served directly from ordinary shared hosting can saturate that host's bandwidth allowance or trigger a resource-limit suspension; a CDN in front of object storage absorbs that instead.

**The honest gap: real adaptive bitrate needs either self-hosted transcoding infrastructure (ffmpeg + HLS/DASH packaging — genuine operational burden, not a WordPress-plugin-sized task) or a purpose-built video-streaming service that does the transcoding FOR you** (Cloudflare Stream, Bunny Stream, Mux are the three worth evaluating — all charge per-minute-stored and per-minute-delivered, not flat object-storage pricing, and all hand back an HLS manifest URL a plain `<video>` tag with `hls.js` can play). This is a materially different, more expensive tier than object storage, and the wizard needs to make that tradeoff legible rather than quietly picking one.

## 2. The tiered recommendation, not a single answer

Matching VISION.md's "scale to the artist's current needs and fiscal limitations" requirement directly — this is not a single "here's the right setup," it's a wizard that asks a few plain questions and recommends one of two real tiers, both fully supported, neither treated as the "lesser" option in the UI copy:

**Tier A — CDN-fronted object storage (the default recommendation for most artists today).** Wraps the ALREADY-INSTALLED Advanced Media Offloader plugin rather than replacing it — this ecosystem's own "own the interface, don't reinvent the third-party piece" posture applies here exactly like it does for WooCommerce. The wizard's job is entirely about making that plugin's setup painless PLUS adding the piece it doesn't cover at all: CDN configuration in front of the bucket (Cloudflare CDN in front of an R2 bucket is the simplest case since both are Cloudflare products with a documented direct integration; CloudFront in front of S3 for AWS; Azure CDN/Front Door in front of Blob Storage). Good range-request-backed seeking, no adaptive bitrate, no transcoding cost — the right default for moderate-length/moderate-resolution video and for anyone cost-sensitive.

**Tier B — a dedicated video-streaming service (Cloudflare Stream as the primary recommendation, since Cloudflare was already named and keeps the artist inside one provider relationship for both tiers).** Real adaptive bitrate, automatic transcoding, thumbnail generation, all handled by the provider — the wizard's job here is account/API-token setup plus wiring bh-streaming's/bh-courses' video step to request an HLS manifest URL instead of a direct file URL, and swapping the front-end `<video>` tag for an HLS-aware player (`hls.js`, MIT-licensed, the standard no-vendor-lock-in choice — native HLS support exists in Safari only, everywhere else needs this). Real, ongoing per-minute cost — the wizard states this plainly with the provider's actual current pricing, not a vague "may cost more."

**The wizard's actual first screen** should be a short, plain-language interview, not a tier picker with jargon: "How long are your typical videos?" / "Do you expect viewers on slow or mobile connections?" / "What's a comfortable monthly budget for video hosting?" — three or four questions, translated into a recommendation ("Tier A should work well for you" / "Given how long your videos run, Tier B will feel noticeably better — here's roughly what it'll cost at your current volume") with a clear "I'll decide for myself" escape hatch straight to either tier's setup, never forcing the interview on someone who already knows what they want.

## 3. The wizard framework itself (reusable beyond media)

This is the part meant to become the shared pattern every future integration wizard copies — worth building as real, reusable own-ur-shit core infrastructure, not a one-off bolt-on inside whichever plugin needs the first one.

Proposed shape, modeled after `OUS_Jobs`/`OUS_Notifications`'s own "one shared mechanism, any plugin plugs in" precedent:

- **`OUS_Wizard`** (new core class) — a step-sequence runner: a wizard is `['id' => ..., 'label' => ..., 'steps' => [ [id, title, render_callback, validate_callback], ... ]]`, registered via a filter (`ous_setup_wizards`, same zero-central-registration shape as `ous_debug_tools`/`bhi_portal_panels`). Handles the actual step-to-step navigation, progress indicator, and persisted in-progress state (so someone can leave mid-wizard and resume — a real, likely-common case for "I need to go find my Cloudflare account first").
- **Per-step contract**: a step's `render_callback` outputs the plain-language question + whatever inputs it needs; its `validate_callback` receives the submitted values and returns either success (advance) or a specific, actionable error string (stay, show why) — this is the enforcement mechanism for "validate in real time, not accept-and-hope" from VISION.md, not just a suggestion left to each wizard author's discipline.
- **A credential-test helper** (`OUS_Wizard::test_connection($type, $credentials)` or similar) that each integration registers a real check against (an actual S3 `HeadBucket` call, an actual Cloudflare API token verification call) — the thing that turns "looks like a valid-shaped API key" into "confirmed this actually works," per VISION.md's real-time-validation rule.
- **Deep-linking to the provider's own dashboard** for each credential-collecting step — a per-provider small config (Cloudflare R2 token creation URL, AWS IAM console deep link, Azure portal Storage Account keys blade) so the step's copy can say "click here" with a real link, not "find your API key" and leave the rest to the person.

First consumer: the media/CDN wizard from Section 2 above. Second, natural consumer once this exists: WooCommerce/payment-gateway setup (already a known pain point for anyone self-hosting commerce), a future email/CalDAV account connection (`ROADMAP-platform-evolution.md`'s `BH_Mail` interface), anything else namable from VISION.md's "big-vision pillars" sections that touches a real external system.

## 4. What's NOT scoped yet, flagged honestly

- The actual wizard UI/JS has zero code written — this doc is the plan, not the implementation.
- Which video-streaming provider is genuinely best (Cloudflare Stream vs. Bunny Stream vs. Mux) needs current, live pricing/API comparison before Tier B's setup steps get built — don't build against assumed pricing.
- Self-hosted transcoding (a "Tier C" for someone who wants adaptive bitrate without a recurring per-minute bill) is explicitly out of scope for now — genuinely heavier infrastructure (ffmpeg worker, HLS packaging, its own storage-growth story) than this wizard framework is meant to carry; worth a fully separate roadmap pass if it ever becomes a real ask, not assumed as a future tier of this same doc.
- Whether `OUS_Wizard` becomes real core infrastructure before or after the media wizard ships is an open sequencing question — building the media wizard as a one-off first and extracting the reusable framework FROM it (rather than designing the framework in the abstract first) is probably the lower-risk order, matching how `BH_Content`/`BH_Commerce` were themselves extracted from real working call sites rather than designed top-down — but that's a call for whoever picks this up next, not decided here.

## 5. Adjacent idea, flagged for later: GitHub-based update checks (AJ, 2026-07-17)

Not media-related, but the same "it just works, no manual step a dev would otherwise have to remember" instinct, and adjacent to infrastructure this ecosystem already has: `OUS_Debug`'s existing "Bundled Zip Freshness" table (Debug Tools → Monitoring & Health) already compares each peer plugin's BUNDLED zip version against its INSTALLED version, so an install running a stale bundled copy gets flagged — but that only catches drift between two copies already sitting on the same server. It says nothing about whether a genuinely NEWER version exists upstream on GitHub that hasn't been pulled down at all yet.

The real ask: check the actual GitHub repo (this one, or any peer plugin's own repo if they're ever split out) for a newer release/tag than what's installed, and surface that in wp-admin — without needing to publish to wordpress.org, which this ecosystem deliberately never will (see VISION.md's no-vendor-lock-in / self-hosted-always posture — a WordPress.org listing is itself a rented, revocable distribution channel).

**This is a solved problem, not one to build from scratch** — per VISION.md's own standing rule ("check for a viable open-source option first"). WordPress core has supported a plugin header (`Update URI: https://github.com/owner/repo`) since 5.8 specifically for self-hosted update sources, and the well-known `plugin-update-checker` library (YahnisElsts/plugin-update-checker, MIT-licensed, ~15k+ installs, no build step — a single vendored PHP file/folder, matching this ecosystem's own SortableJS/FPDF vendoring convention exactly) already handles the harder cases: checking GitHub Releases/tags, showing the real WordPress "Update available" admin notice and one-click update button, and pulling the new zip directly from a GitHub release asset. Reference before building: the library's own GitHub repo and docs.

**Scope, if/when this gets picked up:** one shared `OUS_GithubUpdates`-style registration (same zero-central-registration filter shape as `ous_debug_tools`/`bhi_portal_panels` — a plugin declares its own repo owner/name once) rather than every peer plugin independently vendoring and configuring the update-checker library itself. Real open questions: whether this repo (a monorepo holding every peer plugin, not one-repo-per-plugin) needs each plugin's update check scoped to a subdirectory/tag-prefix rather than the whole repo's latest commit, and whether "latest tag" or "latest commit on a specific branch" (dev vs. a future stable/release branch) is the right freshness signal for an ecosystem that doesn't cut formal releases today. Not decided here — flagged so the idea isn't lost, not scoped enough to build yet.
