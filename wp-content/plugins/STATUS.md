## (a) What this ecosystem actually is

Own Ur Shit is a self-hosted WordPress plugin ecosystem built as "digital civil service infrastructure" — a musician's business-in-a-box that lets someone opt out of renting their audience, data, and tools from big platforms. One required core (`own-ur-shit`) provides shared services (identity, style tokens, jobs/queue, notifications, debug tooling, event tracking, and now a live visual page/element builder), and a set of peer feature plugins — `bh-streaming`, `bh-contest`, `bh-crm`, `bh-registry`, `bh-monetization-woo`, `bh-courses` — each depend only on the core and treat every other peer plugin as optional via `class_exists()` guards. The whole thing runs on ordinary shared hosting (WP-Cron, MySQL, no external services assumed), and the standing architectural discipline is "no vendor lock-in, no quiet dependency on a paid third-party service where an owned equivalent will do." The project owner's current standing priority, restated across nearly every doc, is hardening what exists (core, bh-contest, the style system, bh-crm, Debug Tools) before adding net-new feature surface — though in practice a large amount of net-new infrastructure (the event system, the element/page builder, tier-depth monetization work) has shipped alongside that hardening rather than strictly before it.

## (b) What's built and working, per plugin

**own-ur-shit (core)** is the most actively developed piece, currently at v3.4.57. The original shared-services list — identity/auth with optional TOTP two-factor, the design-token system (`BHY_Style`/`BHY_UI`), a shared abuse-report queue, the install/activation dashboard, Debug Tools, a DB-backed console/log viewer, an in-admin Test Runner, live-generated OpenAPI docs, a notifications inbox, and a WP-Cron-backed job queue — all shipped and are described in VISION.md as production-usable, though never confirmed against a live PHP/MySQL/browser environment (every pass explicitly notes it worked in a walled-off, execution-free sandbox). On top of that, a large "no special-cased pages" initiative has been underway and shipped in real increments: `BH_Content` (a Gutenberg-backed structured content-block interface), `BHI_Portal` (an authenticated `/account/` shell with a pluggable panel system, currently one real panel — profile), `BH_Commerce` (a thin wrapper over WooCommerce product/order calls, migrated for tier-product sync), and — the largest recent push — `BH_Element`, a placement/capability/data-binding system that powers a real three-pane visual builder (Design Suite) unifying the Style editor and an Elementor/Godot-style nested node-tree canvas. That builder now has a real, live-rendered CRM surface, full-subtree prefabs, contextual add-child pickers, and a "Site" root node whose inspector shows global tokens. `BH_Event`/`BH_Identity` (a self-hosted GA/Segment-equivalent event system, table `bhcore_events`, ingested through the job queue) is real and wired into bh-streaming, bh-contest, and bh-courses.

**bh-contest** (v3.1.6) is stable, mature, and mostly in a fix/observability phase now rather than new-feature work — recent versions are Discord logging depth, vote/submission error logging, an auth bug fix on the Live Console dropdown, and wiring the `bh/vote` event emission into the vote/unvote flow.

**bh-courses** (v0.4.8) has moved its lesson authoring off a fixed four-step-type metabox form and onto the `BH_Content`/`BH_Studio` block canvas (per `LMS-AUTHORING-DESIGN-PLAN.md`'s recommendation), added a JSON `answers` snapshot column to `bhc_progress` for quiz review (Part 1 of `QUIZ-AND-CATALOG-DESIGN-PLAN.md`), fixed several real progress-tracking bugs (NULL-vs-placeholder scoring, resubmission overwrite, drip-date timezone mismatch), and split its monolithic renderer into catalog/course/lesson-specific classes.

**bh-crm** (v1.3.5) has been restructured into its own top-level "CRM" admin menu (per `DESIGN-SUITE-UNIFICATION-PLAN.md` Phase 1), gained a real live-rendered element-builder surface for its profile page, collapsed its three framework-defined slots (header/main/sidebar) down to one flexible `root` slot, and now hosts a genuine kanban Project Tracker built entirely on the `BH_Element` system. It is not yet wired to `bhcore_events` — a contact's Activity section does not include pre-signup event history, and this is explicitly named as the next real consumer to build, not a rounding error.

**bh-streaming** (v0.5.2) has its play/skip tracking wired into the new event system (`bhs/play`/`bhs/skip`) alongside its existing daily-stats rollup and per-track counters; a recent pass also fixed a real fatal-risk gap (an unconditional `BHY_Style` call with no `class_exists()` guard) and a recommendations bug that leaked locked/gated tracks and dropped externally-aggregated ones.

**bh-monetization-woo** (v0.4.5) shipped the tier-depth work from `ROADMAP-platform-evolution.md` Section 4 — structured per-tier benefit lists, cover images, annual pricing, and entitlement-granted/revoked action hooks — finished its `BH_Commerce` migration for product sync, and split its previously-overloaded `class-products.php` refund/fraud-pattern detection out into its own `BHM_Fraud` class.

**bh-registry** (v0.1.4) is small and largely quiet — recent activity is logging depth and housekeeping only, no feature growth.

## (c) What's genuinely still open

**Core/design-suite track.** The "no special-cased pages" architecture (collapsing every hand-written admin/front-end template into one node tree rooted at "Site") is a phased, explicitly-not-finished effort: CRM's slots are done, Dashboard/Portal needed no work (they were already single-slot), Gutenberg has a v1 (embedding an existing prefab into a post, no in-canvas authoring yet), and LMS lesson pages are surveyed but not started — `bh-courses` has zero `BH_Element` integration today. `DESIGN-SUITE-PAGE-MANAGER-PLAN.md`'s extension of this to ordinary WP pages/posts (a HubSpot-style page manager, Option B: real `wp_posts` rows with `the_content` fully generated from a node tree) is fully designed but not built — none of its five phases (data model, post-edit metabox, context-picker tree UX, default-on for new pages, delete-cleanup hook) has shipped code yet. `PROJECT-TRACKER-TRACKIT-PARITY-PLAN.md` is explicitly "DESIGN ONLY, NOT BUILT" — reusable checklists, timestamped fixes, a feedback log, stall analytics, and file-linking are all named gaps in bh-crm's kanban tool with a five-phase plan (A–E), none started.

**LMS.** `ROADMAP-lms-v3.md` scopes four bigger ambitions against the shipped block-authoring foundation: interactive video overlays (buildable now, no architecture change needed), continued authoring-tool maturation (ongoing), and two genuinely large ideas — branching lesson paths and a graphic mind-map authoring tool — both explicitly flagged as needing a dedicated design pass before any code, since branching would require reworking `BHC_Progress`'s entire linear-position data model. `QUIZ-AND-CATALOG-DESIGN-PLAN.md`'s Part 2 (course catalog taxonomies, instructor/difficulty/duration fields, popularity sort) has no corresponding changelog entry and should be checked directly against `bh-courses`' post-type/admin code before being treated as either done or open. The original `ROADMAP-feedback-and-courses-v2.md` gap list (admin progress overview, completion hook, quiz retry limits, drip scheduling) has been shipped; what's left from that doc is BH Feedback itself — a whole new peer plugin (paid song feedback: quick-take / detailed / timestamped-audio tiers) — which remains entirely unbuilt, still just a plan, with its reviewer capability (`bhcore_review_submissions`) scaffolded and waiting.

**Monetization/storefront.** Gifting, free trials, and promo codes for supporter tiers don't exist yet. The full storefront/merchandising layer (collection pages, faceted browse/search, merchandising relations) and the auction-listing idea are both roadmap-only, sequenced last in `ROADMAP-platform-evolution.md`'s build order since they depend on `BH_Content` and `BH_Commerce` both being further along.

**Safety/metrics.** `ROADMAP-safety-and-metrics.md`'s near-term list — device+IP refund correlation, purchase-velocity flagging, report-queue prioritization by volume, Jam per-listener mute and participant caps, duplicate-audio-hash detection — is written as a plan, not confirmed shipped by any changelog reviewed here, and should be treated as open until verified against `bh-monetization-woo`/`bh-streaming` code directly. Two items in that doc's own "long-term vision" section are stale: it lists **two-factor authentication** as unbuilt, but `VISION.md` documents `BHI_TwoFactor` as shipped (TOTP, per-account opt-in, enforced on both wp-login and the REST login endpoint) — the safety/metrics roadmap needs updating, not the vision doc. Similarly, that roadmap's "near-term" Item 1 (an artist-facing aggregate metrics dashboard for bh-streaming) is described as not-yet-built plan content, while `VISION.md` refers to bh-streaming as already having "its own aggregate metrics dashboard (plays, skip-rate, geo/referrer)" — these two docs disagree on whether that dashboard exists; this should be resolved by checking `bh-streaming`'s admin pages directly rather than trusting either doc as-is.

**Cross-cutting/platform-evolution gaps**, all confirmed still open by both the roadmap docs and the absence of contrary changelog evidence: a real role-assignment admin UI (capabilities can only be granted via one-off code or a filter today), per-notification-type email preferences, a front-end user bar for non-admin users, a rolled-up cross-plugin metrics dashboard (`bhcore_metrics_widgets`), all of marketing tooling (email broadcast/campaigns, referral/affiliate tracking, link-in-bio landing pages, SEO basics), `BH_Media`/`BH_Mail` interfaces (both explicitly low-priority, deferred), and the entire social/marketing integration category (YouTube, Twitch, Meta, TikTok, Roku Ads) which stays roadmap-only by explicit direction. The three "big vision pillars" (collaborative docs/files, a Canva-style design tool, and Jira-style project management) and the "round two" pillars (BI/reporting, ERP/accounting, tax tooling, and email/calendar as CalDAV/IMAP-based Workspace/365 replacements) remain unscoped, undecided strategic bets — VISION.md itself frames the choice of whether to pursue any of them as a deliberate decision still owed, not yet made.
